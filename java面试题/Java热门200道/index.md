## 1. MyBatis 中 #{} 和 ${} 的区别是什么？

🟢 **简单** | 标签：后端, MyBatis

在MyBatis中，`#{}` 和 `${}` 都用于SQL语句的参数绑定，但核心区别在于处理机制和安全性。

`#{}` 是**预编译处理**。MyBatis会将其替换为JDBC中的占位符 `?`，通过PreparedStatement设置参数值。这种方式能有效防止SQL注入攻击，因为参数会被安全地转义和处理，适用于大多数参数值场景。

`${}` 是**字符串替换**。MyBatis直接将参数值拼接到SQL语句中，不做预编译。这会导致SQL注入风险，因为恶意输入可能被直接执行。然而，`${}` 适用于需要动态SQL片段的场景，如表名、列名或ORDER BY子句，因为这些位置无法使用占位符。

总结：在编写SQL时，应优先使用 `#{}` 保证安全性；仅当必须动态拼接SQL结构时才考虑 `${}`，并确保参数来源可信。

## 2. MySQL 中 count(*)、count(1) 和 count(字段名) 有什么区别？

🟢 **简单** | 标签：后端, MySQL, 数据库

MySQL中`count(*)`、`count(1)`和`count(字段名)`的核心区别在于**计数的范围和语义**。

1.  **`count(*)` 与 `count(1)`：统计总行数**
    在标准SQL中，它们的含义是**等价的**，都是用于统计**表中满足查询条件的总行数**。MySQL优化器会将它们同样处理，不会去解析具体的列值。它们会包含所有行，无论字段值是否为`NULL`。

2.  **`count(字段名)`：统计指定字段的非NULL行数**
    这个写法会统计**指定字段中值不为`NULL`的行数**。如果该字段存在`NULL`值，这些行将不被计入。因此，`count(*)` 的结果可能大于或等于 `count(字段名)` 的结果。

**性能与选择：**
*   在InnoDB引擎下，`count(*)` 和 `count(1)` 性能基本相同，且通常最优。优化器会选择最小的可用索引来扫描计数。
*   `count(字段名)` 的性能可能稍差，特别是当该字段没有索引时，需要进行额外的判断（检查是否为NULL）。
*   **建议**：若要统计总行数，应首选`count(*)`，这是SQL标准且意图最清晰。若需统计某列非空值的个数，则使用`count(字段名)`。

**总结**：`count(*)` 计总行，`count(列)` 计该列非空值的行数。

## 3. MySQL 中的事务隔离级别有哪些？

🟢 **简单** | 标签：后端, MySQL, 数据库

MySQL 中的事务隔离级别定义了事务之间数据可见性的规范，旨在平衡并发性能与数据一致性。主要包含以下四种：

1.  **读未提交**：最低级别。事务可以看到其他事务尚未提交的修改。这会导致**脏读**（读取到可能回滚的数据）、不可重复读和幻读。
2.  **读已提交**：事务只能看到其他事务已提交的数据。解决了脏读，但同一事务内多次读取同一数据，可能得到不同结果（**不可重复读**），且仍可能出现幻读。
3.  **可重复读**：**MySQL 默认级别**。保证在同一事务内多次读取同一数据的结果是一致的，解决了不可重复读。通过多版本并发控制等技术，很大程度上避免了幻读，但理论上并未完全解决。
4.  **串行化**：最高级别。强制事务串行执行，完全避免了脏读、不可重复读和幻读，但并发性能最差。

**核心要点**：隔离级别由低到高，并发能力递减，数据一致性递增。选择时需根据业务场景对一致性和性能的要求进行权衡。MySQL 默认使用“可重复读”级别，能够在大多数场景下兼顾一致性与性能。

## 4. MySQL 的索引类型有哪些？

🟢 **简单** | 标签：后端, MySQL, 数据库

MySQL 中常见的索引类型主要包括以下几种：

1. **B+Tree 索引**：这是最常用的索引类型，尤其在 InnoDB 存储引擎中。它适用于全键值、键值范围或键前缀查找，能够高效支持排序和分组操作。InnoDB 的主键索引（聚簇索引）将数据和索引存储在一起，而非主键索引（二级索引）的叶子节点存储主键值。

2. **Hash 索引**：基于哈希表实现，仅支持精确匹配查询，不支持范围查询。在 Memory 存储引擎中支持。查询速度快，但仅适用于等值查询（如 `=`、`IN`）。

3. **全文索引（Full-Text Index）**：用于在文本数据（如 `CHAR`、`VARCHAR`、`TEXT`）中进行关键词搜索。InnoDB 和 MyISAM 引擎支持。它通过分词技术提高搜索效率，适用于 `MATCH AGAINST` 操作。

4. **空间索引（Spatial Index）**：用于地理空间数据（如 `GEOMETRY`、`POINT`）。MyISAM 和 InnoDB（MySQL 5.7+）支持，主要用于 GIS 应用中的空间数据查询。

5. **前缀索引**：对字符串的前几个字符建立索引，适用于长字符串列，可减少索引大小，但无法用于 ORDER BY 或 GROUP BY。

6. **联合索引（复合索引）**：在多个列上建立的索引，遵循最左前缀原则。查询时只有从索引最左列开始连续使用，索引才会生效。

## 5. MySQL 索引的最左前缀匹配原则是什么？

🟢 **简单** | 标签：后端, MySQL, 数据库

MySQL索引的最左前缀匹配原则是指，在使用联合索引时，查询条件必须从索引的最左列开始，并且是连续匹配，才能有效利用该索引。

**核心原理**：联合索引在B+树中的数据结构，是按照索引定义的列顺序从左到右依次排序的。只有满足最左前缀的查询条件，才能利用这种有序性进行快速检索。

**具体规则**：
1.  **全列匹配**：查询条件精确匹配索引的全部列，索引效率最高。
2.  **最左前缀连续匹配**：查询条件从索引的第一列开始，并且是连续的。例如，索引为`(a, b, c)`，查询条件为`WHERE a=1 AND b=2`可以命中索引。
3.  **查询跳跃最左列，索引部分失效**：查询条件未从第一列开始，或中间有缺失，则无法利用该联合索引。例如，直接使用`WHERE b=2`或`WHERE a=1 AND c=3`（跳过了b），通常无法高效利用`(a, b, c)`索引。
4.  **范围查询后的列失效**：如果查询条件中对某一列使用了范围查询（如`>`、`<`、`BETWEEN`），则该列右边的所有索引列将无法用于加速查询。

**实际应用**：设计联合索引时，应将选择性高（区分度大）、最常出现在查询条件中的列放在最左侧。理解并遵循此原则，是编写高效SQL和优化数据库性能的关键。

## 6. MySQL 默认的事务隔离级别是什么？为什么选择这个级别？

🟢 **简单** | 标签：后端, MySQL, 数据库

MySQL 默认的事务隔离级别是“可重复读”（REPEATABLE READ）。这一选择基于数据一致性与性能平衡的考量。

事务隔离级别定义了多个事务并发时的数据可见性，MySQL 提供了四种标准级别：读未提交、读已提交、可重复读和串行化。可重复读作为默认级别，核心优势在于它能有效防止脏读和不可重复读问题，同时允许较高的并发性能。在 MySQL 的 InnoDB 存储引擎中，通过多版本并发控制（MVCC）和行级锁机制实现这一级别，在保证事务看到一致数据快照的前提下，减少锁竞争，提升吞吐量。

相比于更严格的串行化级别，可重复读避免了过度加锁导致的性能下降；而相比于读已提交，它提供了更强的数据一致性保障，适合大多数需要事务支持的应用场景，如电商、金融系统等。因此，MySQL 选择可重复读作为默认设置，在数据完整性与系统效率之间取得了合理平衡。

## 7. Redis 中常见的数据类型有哪些？

🟢 **简单** | 标签：后端, Redis

Redis 提供了丰富的数据类型，主要分为基础类型和扩展类型：

**基础数据类型：**
1. **String**：最基础的类型，可存储字符串、整数或浮点数，常用于缓存、计数器、分布式锁。
2. **List**：双端链表，支持队列、栈操作，适用于消息队列、最新列表等场景。
3. **Hash**：键值对集合，适合存储对象（如用户信息），比多个String更节省内存。
4. **Set**：无序且去重的集合，支持交集、并集、差集运算，可用于共同关注、标签系统。
5. **Sorted Set (ZSet)**：带分数的有序集合，可按分值排序，常用于排行榜、带权重的任务队列。

**扩展数据类型：**
- **Bitmap**：位操作，适合统计用户签到、活跃状态等二值场景。
- **HyperLogLog**：基数估算，以极低内存统计UV等大数据量去重计数。
- **Geo**：地理位置，支持距离计算、范围查找，用于附近的人等功能。
- **Stream**：类似消息队列，支持消费者组和持久化，适用于事件流处理。

选择合适的数据类型能极大提升性能并降低复杂度。

## 8. Redis 通常应用于哪些场景？

🟢 **简单** | 标签：后端, Redis

Redis 在后端开发中应用广泛，主要得益于其高性能、丰富的数据结构和原子操作。核心应用场景包括：

1.  **缓存**：作为数据库前的高速缓存，缓存热点数据（如用户信息、商品详情），大幅降低数据库负载，提升系统响应速度。
2.  **分布式锁**：利用其原子操作（如 `SETNX`）实现跨进程的互斥控制，确保分布式环境下资源访问的准确性，常用于秒杀、订单处理等场景。
3.  **会话管理**：存储用户的 Session 信息，支持集群环境下的会话共享与快速存取。
4.  **排行榜/计数器**：借助有序集合（Sorted Set）实现实时排名（如游戏积分榜），使用 `INCR` 等命令高效实现页面访问量、点赞数统计。
5.  **消息队列**：通过列表（List）或发布订阅功能实现简单的消息缓冲或实时通知，适用于对可靠性要求不高的异步解耦场景。
6.  **限流器**：利用其原子操作和过期机制，实现API访问频率控制，保护系统稳定。

总结而言，Redis 的核心价值在于解决高性能读写、数据结构复杂操作及分布式协调问题，是构建高并发、低延迟系统的重要组件。

## 9. TCP 和 UDP 有什么区别？

🟢 **简单** | 标签：网络

**TCP与UDP的核心区别**

TCP（传输控制协议）和UDP（用户数据报协议）是传输层的两种核心协议，主要区别如下：

1.  **连接性**：TCP是**面向连接**的，传输前需通过“三次握手”建立可靠连接，结束后需“四次挥手”释放连接。UDP是**无连接**的，发送数据前无需建立连接。

2.  **可靠性**：TCP提供**可靠传输**。它通过序列号、确认应答、重传机制等确保数据无差错、不丢失、不重复且按序到达。UDP则提供**不可靠传输**，它仅尽最大努力交付，不保证数据完整性和顺序。

3.  **传输效率**：TCP因复杂的连接管理、拥塞控制和流量控制机制，传输效率相对较低，有延迟。UDP**头部开销小（仅8字节）**，无额外控制，传输效率高，延迟低。

4.  **数据形式**：TCP是**面向字节流**的，数据被视为一连串无结构的字节序列。UDP是**面向报文**的，保留报文边界，应用层交付多大，UDP就发送多大。

5.  **应用场景**：TCP适用于对**可靠性要求高**的场景，如网页浏览（HTTP/HTTPS）、文件传输（FTP）、电子邮件（SMTP）等。UDP适用于对**实时性要求高、能容忍少量丢包**的场景，如视频直播、在线游戏、语音通话、DNS查询等。

**总结**：TCP是“可靠但慢”，UDP是“快速但不可靠”。在实际应用中，应根据业务对可靠性、实时性的具体需求进行选择。

## 10. TCP 是用来解决什么问题？

🟢 **简单** | 标签：网络

TCP（传输控制协议）主要用于解决IP网络中**数据传输的可靠性、顺序性和流量控制**问题。IP协议本身是“尽力而为”的，不保证数据包能完整、有序到达。TCP通过以下机制解决这些问题：

1.  **可靠传输**：通过序列号、确认应答（ACK）和超时重传机制，确保数据不丢失、不重复。
2.  **顺序控制**：为每个数据包编号，接收方能按发送顺序重组数据。
3.  **流量控制**：使用滑动窗口机制，让接收方告知发送方其处理能力，防止发送过快导致接收方缓冲区溢出。
4.  **拥塞控制**：通过慢启动、拥塞避免等算法，感知网络状况并动态调整发送速率，避免加重网络拥堵。

简言之，TCP在不可靠的IP网络之上，建立了一个**面向连接的、可靠的、字节流**通信服务，为应用层（如HTTP、FTP）提供了稳定的数据传输基础。

## 11. TCP 超时重传机制是为了解决什么问题？

🟢 **简单** | 标签：网络

TCP超时重传机制是保障可靠传输的核心机制之一，主要解决**数据包在传输过程中可能丢失或损坏**的问题，确保数据能最终完整送达。

在网络传输中，数据被分割为报文段发送。若发送方发出某个报文段后，在特定时间内（称为**超时时间**）未收到接收方的确认（ACK），则默认该报文段已丢失或传输失败。此时发送方将自动重传该报文段，直至成功接收确认。

该机制的关键在于**超时时间（RTO）的设定**：
1.  **基于往返时间（RTT）估算**：RTO会根据网络往返时延动态调整，避免因网络延迟误判为丢失，或因过长等待降低效率。
2.  **指数退避策略**：若重传多次仍失败，RTO会按指数级增长，避免在网络拥塞时加剧负担。

因此，超时重传机制通过**时间监控和自适应重试**，解决了因网络不可靠性导致的数据丢失问题，是TCP实现**端到端可靠传输**的重要基础。在面试中，可结合网络不确定性、ACK确认、RTO动态计算等要点进行阐述。

## 12. TCP/IP 四层模型是什么？

🟢 **简单** | 标签：网络

TCP/IP 四层模型是互联网通信的基础框架，它将网络通信过程划分为四个功能层，自上而下分别为：

1.  **应用层**：为用户提供具体的网络服务，负责处理特定的应用程序细节。常见的协议如HTTP、FTP、SMTP、DNS等都工作在此层。
2.  **传输层**：提供端到端的通信服务，负责数据流的分段、重组和错误控制。核心协议是面向连接的TCP（保证可靠性）和无连接的UDP（追求效率）。
3.  **网络层**：负责数据包的路由和转发，实现不同网络之间的互连。核心协议是IP协议，它为每一台主机分配地址（IP地址），并决定数据包的最佳传输路径。
4.  **网络接口层**：也称为链路层，负责处理与物理网络媒介（如以太网、Wi-Fi）的接口，将数据包封装成帧进行传输，并进行物理寻址（MAC地址）。

**核心要点**：当数据从应用层发送时，会自上而下依次经过各层的封装，每层添加自己的头部信息。到达目的地后，再自下而上逐层解封装，最终将数据交付给目标应用。这个模型清晰地定义了各层的职责，实现了协议和实现的解耦，是理解网络通信的关键。

## 13. 什么是分库分表？分库分表有哪些类型（或策略）？

🟢 **简单** | 标签：后端, MySQL, 数据库

分库分表是应对海量数据和高并发场景的数据库架构优化策略，主要解决单库单表容量和性能瓶颈问题。

**核心类型分为两种：**

1.  **垂直拆分**
    *   **垂直分库**：按业务模块将不同业务的表拆分到独立数据库中（如订单库、用户库），实现业务解耦，便于独立扩展和管理。
    *   **垂直分表**：将一张包含大字段或访问频率差异大的表，按列拆分成多张表（如将商品详情表拆分为主表和详情表），减少单行数据量，提升查询效率。

2.  **水平拆分**
    *   **水平分库**：将同一个业务的数据，按某个字段（如用户ID）的哈希或范围规则，分散到多个结构相同的数据库中，用于分散单库的数据量和访问压力。
    *   **水平分表**：将同一个业务表的数据，按分片键（如订单ID）拆分到多个结构相同的表中，解决单表数据量过大的问题。

**实际应用**中常组合使用，例如先按业务垂直分库，再对某些大表进行水平拆分。实施时需借助中间件（如ShardingSphere）来路由SQL和聚合结果，并需妥善处理分布式事务、跨库关联等挑战。

## 14. 什么是服务雪崩？

🟢 **简单** | 标签：服务容灾, Spring Cloud, 微服务, 后端

服务雪崩是指在微服务架构中，一个基础服务的故障如同雪崩一样，引发依赖它的多个上层服务依次失效，最终导致整个分布式系统不可用的连锁反应。

其核心原因是服务间的依赖调用形成了一条“故障链”。例如，当一个核心服务（如用户服务）因数据库连接耗尽而响应缓慢时，所有调用它的服务（如订单、商品服务）的线程池会逐渐被阻塞的请求占满。这些服务也随之变得不可用，进而导致依赖它们的更多服务崩溃，故障像滚雪球一样蔓延。

为解决此问题，主要采用服务容灾策略：
1.  **服务熔断**：当下游服务错误率超阈值时，主动“切断”电路，快速返回失败或降级结果，避免资源耗尽。
2.  **限流与降级**：通过限制瞬时流量保护服务，并制定降级预案，在系统压力过大时暂时关闭非核心功能。
3.  **超时控制与健康检查**：设置合理的调用超时，及时剔除不健康的服务实例。

总之，服务雪崩是分布式系统级联故障的典型表现，核心应对思路是通过**快速失败**和**资源隔离**，防止局部故障扩散为全局灾难。

## 15. 什么是模板方法模式？一般用在什么场景？

🟢 **简单** | 标签：设计模式

**模板方法模式**是一种行为型设计模式，其核心是：**在父类中定义一个算法的骨架（模板方法），将某些步骤的具体实现延迟到子类中，从而在不改变算法结构的前提下，让子类重定义算法的某些特定步骤。**

**核心要点与结构：**
1.  **抽象类（Abstract Class）**：负责定义算法骨架，即模板方法。该方法通常由一系列步骤组成。
2.  **模板方法**：定义在抽象类中，通常为`final`，以防止子类覆写算法的整体流程。它调用若干基本方法。
3.  **基本方法**：
    *   **抽象方法**：必须由子类实现的步骤。
    *   **具体方法**：在父类中已实现的通用步骤。
    *   **钩子方法**：在父类中提供默认实现（通常为空），子类可选择性覆盖，以影响算法流程。

**典型应用场景：**
1.  **框架设计**：框架定义整体处理流程（如生命周期），将具体实现（如初始化、业务逻辑）交给使用者扩展。例如，Servlet的`doGet/doPost`。
2.  **数据处理**：定义数据处理的固定流程（如“读取数据 -> 处理 -> 保存”），但读取源和保存方式由子类决定。
3.  **生命周期管理**：如测试框架中，`setUp()`和`tearDown()`作为钩子，由测试子类实现具体准备和清理工作。

**优势**：复用代码，约束子类行为，符合开闭原则。**缺点**：子类数量可能增多，增加系统复杂性。

## 16. 什么是策略模式？一般用在什么场景？

🟢 **简单** | 标签：设计模式

策略模式是一种行为设计模式，它将定义的一系列算法封装起来，并使它们可以相互替换，让算法独立于使用它的客户端而变化。

其核心思想是“**面向接口编程**”。策略模式通常包含三个角色：**Strategy（抽象策略接口）**，定义算法的公共接口；**ConcreteStrategy（具体策略类）**，实现具体的算法；**Context（上下文类）**，持有策略接口的引用，并负责与客户端的交互。

这种模式主要应用于需要**动态切换算法或行为**的场景。典型应用包括：
1.  **电商折扣策略**：如满减、折扣、返券等不同促销规则，可在结算时动态切换。
2.  **支付方式选择**：在订单支付时，可灵活选择支付宝、微信、银联等不同支付算法。
3.  **文件压缩/导出格式**：根据用户选择使用不同的压缩算法（如ZIP、RAR）或导出为不同格式（如PDF、Excel）。
4.  **替代多重条件判断**：当代码中存在大量`if-else`或`switch`语句来处理不同行为时，使用策略模式可将其重构为可插拔的策略类，显著提升代码的扩展性和可维护性。

总而言之，策略模式通过分离算法的定义与使用，实现了“开闭原则”，让系统更易于扩展和维护。

## 17. 什么是观察者模式？一般用在什么场景？

🟢 **简单** | 标签：设计模式

观察者模式是一种行为型设计模式，它定义了对象间一对多的依赖关系。当一个对象（称为“主题”或“可观察者”）的状态发生改变时，所有依赖于它的对象（称为“观察者”）都会自动得到通知并更新。

其核心角色包括：
1.  **Subject（主题）**：维护一个观察者列表，提供添加、移除和通知观察者的方法。
2.  **Observer（观察者）**：定义一个更新接口，在得到通知时进行自我更新。
3.  **ConcreteSubject/ConcreteObserver**：具体实现。

**工作原理**：观察者向主题注册自身。当主题内部状态变化时，会遍历其观察者列表，调用每个观察者的更新方法，将变化通知出去。这实现了主题与观察者之间的松耦合，便于独立扩展和复用。

**典型应用场景**：
1.  **事件处理系统**：如GUI框架中，按钮（主题）被点击后通知所有注册的事件监听器（观察者）。
2.  **消息订阅/发布**：消息中间件、新闻订阅等，发布者更新内容后，所有订阅者自动接收。
3.  **模型-视图架构（MVC）**：当模型（Model）数据变化时，多个视图（View）可以自动更新显示。
4.  **Spring中的事件机制**：ApplicationEventPublisher发布事件，多个ApplicationListener进行监听和处理。

## 19. 什么是责任链模式？一般用在什么场景？

🟢 **简单** | 标签：设计模式

责任链模式是一种行为设计模式，其核心思想是将请求的发送者与接收者解耦，使多个对象都有机会处理该请求。这些对象被连接成一条链，并沿着这条链传递请求，直到有一个对象处理它为止。该模式包含两个关键角色：处理者（定义处理请求的接口）和具体处理者（实现处理逻辑，并持有对下一个处理者的引用）。

典型应用场景包括：
1. **审批流程**：如企业中的请假、报销申请，根据金额或级别需经过部门经理、总监、总经理等层层审批。
2. **Web过滤器**：在Servlet中，多个过滤器组成链，依次对请求进行编码、权限校验、日志记录等预处理。
3. **日志处理**：根据日志级别（DEBUG、INFO、ERROR）将其传递给不同的处理器（控制台、文件、远程服务器）。

该模式的优势在于降低了耦合度，增强了给对象分配职责的灵活性，可通过动态调整链内成员来改变处理流程。但也需注意避免过长的链条影响性能，以及确保请求最终能被妥善处理。

## 21. 到底什么是 TCP 连接？

🟢 **简单** | 标签：网络

TCP连接是网络中可靠的、面向连接的通信链路，通过**三次握手**建立、**四次挥手**断开，确保数据有序、无差错传输。其核心特征包括：

1.  **可靠性**：通过序列号、确认应答、超时重传、校验和机制保障数据完整送达。
2.  **面向连接**：通信前需建立虚拟电路，双方维持连接状态（如`ESTABLISHED`）。
3.  **全双工通信**：连接建立后，双方可同时收发数据。
4.  **流量与拥塞控制**：滑动窗口机制优化传输效率，避免网络拥塞。

建立过程以**三次握手**为例：客户端发送`SYN` → 服务端回复`SYN+ACK` → 客户端发送`ACK`，双方同步初始序列号并确认连接可用。此设计解决了网络延迟导致的重复连接请求问题。

在Java中，`Socket`和`ServerSocket`即是对TCP连接的封装。理解TCP连接是掌握网络编程与性能调优的基础。

## 22. 在 MySQL 中建索引时需要注意哪些事项？

🟢 **简单** | 标签：后端, MySQL, 数据库

在MySQL中创建索引时，核心注意事项如下：

1.  **列的选择**：优先在**高选择性**（如用户ID、订单号）、**高频查询**（WHERE、JOIN、ORDER BY子句常用）的列上建索引。避免在**低选择性**（如性别）或**频繁更新**的列上过度索引。

2.  **数量与权衡**：索引并非越多越好。每个索引都会占用存储空间，并**显著降低INSERT、UPDATE、DELETE等写操作的性能**，因为需要维护索引结构。应在**查询速度**和**写入性能**之间取得平衡。

3.  **复合索引设计**：
    *   **最左前缀原则**：复合索引(a, b, c)能支持查询a、(a,b)或(a,b,c)，但无法直接支持查询b或c。
    *   **顺序很重要**：将选择性最高、查询最频繁的列放在最左侧。
    *   **覆盖索引**：尽量让索引包含查询所需的所有字段，避免回表查询，极大提升性能。

4.  **类型与长度**：为**长字符串**列建索引时，可指定前缀长度（如`INDEX(col(10))`）以节省空间。根据查询模式选择合适索引类型，如B+树用于范围查询，哈希索引用于等值查询。

5.  **定期优化**：使用`EXPLAIN`分析查询计划，检查索引是否被有效利用。定期删除无用或重复的索引，并处理索引碎片以维持性能。

## 23. 说说 TCP 的三次握手？

🟢 **简单** | 标签：网络

TCP三次握手是建立可靠连接的核心机制，其过程如下：

1. **第一次握手**：客户端向服务器发送SYN报文（SYN=1, seq=x），进入SYN_SENT状态，请求建立连接。
2. **第二次握手**：服务器收到请求后，回复SYN+ACK报文（SYN=1, ACK=1, seq=y, ack=x+1），进入SYN_RCVD状态，确认客户端请求并同步自身序列号。
3. **第三次握手**：客户端收到后发送ACK报文（ACK=1, seq=x+1, ack=y+1），双方进入ESTABLISHED状态，连接建立完成。

**核心要点**：
- **序列号同步**：双方通过交换初始序列号（ISN）确保后续数据有序传输。
- **防失效连接**：三次握手避免历史重复连接请求导致服务器错误建立无效连接。
- **状态保障**：每一步均涉及状态变迁，确保双方均具备收发能力后才传输数据。

该设计平衡了可靠性与效率，是TCP实现可靠通信的基础。

## 24. 说说 TCP 的四次挥手？

🟢 **简单** | 标签：网络

TCP四次挥手是连接释放的过程：

1. **第一次挥手**：客户端发送FIN报文（FIN=1，seq=u），进入FIN_WAIT_1状态，表示数据发送完毕，请求关闭连接。
2. **第二次挥手**：服务端收到FIN后，发送ACK报文（ACK=1，ack=u+1），进入CLOSE_WAIT状态。客户端收到ACK后进入FIN_WAIT_2状态。此时连接处于半关闭状态，服务端仍可发送数据。
3. **第三次挥手**：服务端数据发送完毕后，发送FIN报文（FIN=1，seq=w），进入LAST_ACK状态，请求关闭连接。
4. **第四次挥手**：客户端收到FIN后，发送ACK报文（ACK=1，ack=w+1），进入TIME_WAIT状态，等待2MSL（最大报文段生存时间）后关闭。服务端收到ACK后立即关闭。

**核心要点**：
- 需要四次是因为TCP全双工，需双向单独关闭。
- TIME_WAIT状态确保最后ACK可靠到达，避免旧连接数据干扰新连接。
- 服务端可能因数据未发完，延迟关闭连接。

此机制保证了连接可靠释放。

## 25. 请描述简单工厂模式的工作原理。

🟢 **简单** | 标签：设计模式

简单工厂模式是一种创建型设计模式，它通过一个工厂类来封装对象的创建逻辑，客户端无需知道具体创建细节，只需通过工厂传递参数来获取所需对象。

其核心包含三个角色：工厂类、抽象产品接口和具体产品类。工厂类包含一个创建产品的方法，根据客户端传入的参数（如类型标识），决定实例化哪一个具体产品子类。这些具体产品类都实现了共同的抽象产品接口，保证了接口的统一。

## 27. HTTP 1.0 和 2.0 有什么区别？

🟡 **中等** | 标签：网络

HTTP 2.0 相较于 1.0 是一次重大性能升级，核心区别在于传输模型和效率。HTTP 1.0 基于文本，存在**队头阻塞**，即一个连接上同一时刻只能处理一个请求，后续请求必须等待，导致性能低下。而 HTTP 2.0 通过以下关键改进彻底解决了这些问题：

1.  **二进制分帧与多路复用**：将通信数据拆分为更小的二进制帧，并支持在一个 TCP 连接上**并发交错地发送和接收多个请求/响应**（流），彻底消除了应用层的队头阻塞。
2.  **头部压缩**：使用 HPACK 算法压缩重复的 HTTP 头部，显著减少了协议开销，尤其对移动网络和小数据包请求提升明显。
3.  **服务端推送**：允许服务器在客户端请求主资源后，**主动预测并推送**关联的资源（如 CSS、JS），减少了往返延迟。

总之，HTTP 2.0 通过**多路复用**提升了并发效率，通过**头部压缩**减少了冗余数据，并引入了**服务端推送**，从而实现了更低的延迟和更高的吞吐量，是现代高性能 Web 应用的基础。

## 28. HTTP 2.0 和 3.0 有什么区别？

🟡 **中等** | 标签：网络

HTTP 2.0 与 3.0 的核心区别在于底层传输协议的根本性变革，主要体现在以下几点：

1.  **传输层协议**：HTTP/2 基于 **TCP** 传输，而 HTTP/3 彻底转向 **基于 UDP 的 QUIC 协议**。这是最本质的区别。
2.  **队头阻塞**：HTTP/2 虽在应用层实现了多路复用，但 TCP 层的队头阻塞问题依然存在（单个 TCP 丢包会阻塞所有流）。HTTP/3 通过 QUIC 在**传输层原生支持多路复用**，彻底解决了此问题。
3.  **连接建立速度**：HTTP/2 需要 TCP 三次握手 + TLS 握手，耗时较长。HTTP/3 的 QUIC 协议将**传输层握手与加密握手合并**，能实现 **0-RTT 或 1-RTT 的快速连接建立**，尤其对首次访问提升显著。
4.  **加密方式**：HTTP/2 中 TLS 可选，但 HTTP/3 **强制要求使用 TLS 1.3 加密**，安全性更高。
5.  **连接迁移**：基于 TCP 的 HTTP/2 连接由四元组标识，网络切换（如 Wi-Fi 转 4G）会导致连接中断。QUIC 使用**连接ID**，可实现**无缝连接迁移**，极大优化移动网络体验。

总结来说，HTTP/3 的核心目标是解决 HTTP/2 在复杂网络环境下的性能瓶颈，通过 QUIC 协议提供更快、更稳定、更安全的连接体验。

## 29. HTTP 与 RPC 之间的区别？

🟡 **中等** | 标签：微服务, Spring Cloud, 远程调用, 后端

HTTP 和 RPC 的核心区别在于，HTTP 是一种通用的应用层通信协议，而 RPC 是一种远程过程调用的思想与框架。在微服务实践中，两者常被结合使用。

具体来说：

1.  **协议层级与目标**：HTTP（如RESTful）是通用的Web协议，关注资源表述和状态转移。RPC框架（如Dubbo、gRPC）目标是让远程调用像本地方法一样简单，更聚焦于服务间的高效通信。

2.  **调用方式与效率**：HTTP客户端需手动处理URL、HTTP方法和序列化。RPC客户端通常通过接口代理直接调用，框架底层处理通信，对开发者更透明。在性能上，许多RPC框架采用二进制协议（如Protobuf、Hessian），传输效率和序列化性能通常优于基于文本的JSON over HTTP。

3.  **适用场景**：HTTP因通用、简单，适合对外公开的API或需要跨语言、跨平台的Web集成。RPC因性能高、调用直观，更适用于企业内部微服务间的高性能、强类型化调用。

在Spring Cloud生态中，Feign等工具通过封装HTTP客户端，也提供了近似RPC的开发体验。选择时需权衡通用性、性能及团队技术栈。

## 30. HTTP 和 HTTPS 有什么区别？

HTTP（超文本传输协议）和HTTPS（安全超文本传输协议）都是用于Web通信的协议，主要区别在于安全性。HTTPS在HTTP基础上增加了SSL/TLS加密层，确保数据传输的机密性和完整性。  
核心区别包括：  
1. **安全性**：HTTP以明文传输数据，易受窃听和篡改；HTTPS通过加密保护数据，防止中间人攻击。

4. **性能**：HTTPS因加密过程可能引入轻微延迟，但现代协议（如HTTP/2）优化了性能。  
5. **应用场景**：HTTP适用于非敏感内容，HTTPS则用于登录、支付等敏感操作，是当今Web安全的标准。  
在面试中，可强调HTTPS的重要性，尤其随着隐私保护意识的提升，它已成为网站必备。

## 31. Kafka 中  Zookeeper 的作用？

🟡 **中等** | 标签：Kafka, 消息队列, 后端

Zookeeper 在 Kafka 架构中扮演着核心的元数据管理和集群协调角色，其主要作用如下：

**1. 集群元数据存储与管理**
Zookeeper 维护着 Kafka 集群的关键元数据，包括所有 Broker 的注册信息、Topic 与 Partition 的分配详情、以及每个 Partition 的 Leader/Follower 副本所在的 Broker 位置。这为集群提供了统一的、可靠的状态信息视图。

**2. Controller 选举与故障转移**
当 Kafka 集群启动时，多个 Broker 会通过在 Zookeeper 上创建临时节点来竞争成为 Controller。Controller 负责管理分区的 Leader 选举和副本状态机。当当前 Controller 失效时，Zookeeper 能够通过临时节点机制快速感知并触发新的 Controller 选举，保障集群的高可用。

**3. 生产者与消费者协调**
- **生产者**：在发送消息前，会从 Zookeeper（或通过向 Broker 请求）获取 Topic 的分区 Leader 信息，以确定消息发送目标。
- **消费者**：消费者组（Consumer Group）的成员管理和 Offset（消费进度）存储在早期版本中完全依赖 Zookeeper。虽然新版本 Kafka 将 Offset 管理迁移到了内部 Topic `__consumer_offsets`，但消费者组的协调（如分区分配策略）仍可能涉及 Zookeeper。

**补充说明**：随着 Kafka 的发展，社区正逐步推动减少对 Zookeeper 的依赖，通过 KIP-500 等提案旨在实现完全自管理的集群元数据，以提升架构简洁性和可维护性。但在当前主流生产环境中，理解 Zookeeper 的核心作用仍至关重要。

## 32. Kafka为什么要抛弃  Zookeeper？

🟡 **中等** | 标签：Kafka, Zookeeper, 消息队列

Kafka 抛弃 Zookeeper 的主要原因是为了**简化架构、提升可扩展性和可靠性**。

核心原因包括：
1.  **扩展瓶颈**：Zookeeper 集群管理大量 Kafka 分区元数据时，其读写性能成为系统瓶颈，限制了 Kafka 集群的进一步扩展。
2.  **运维复杂**：依赖 Zookeeper 意味着需要独立部署、监控和维护一个分布式系统，增加了运维成本和故障点。
3.  **稳定性风险**：Zookeeper 的会话超时、网络分区等问题会直接影响 Kafka 集群的可用性和控制器选举。

因此，Kafka 引入了 **KRaft（Kafka Raft）** 模式，将元数据管理内化：
*   **元数据自管理**：使用内置的 Raft 共识协议，由 Kafka 自身的控制器（Controller）节点直接管理集群元数据和领导选举。
*   **简化部署与运维**：无需再依赖外部系统，降低了架构复杂度和运维负担。
*   **提升性能与规模**：消除了外部瓶颈，能够支持更大规模的分区（百万级），并缩短了控制器故障恢复时间。

目前，KRaft 模式已成为 Kafka 的默认推荐模式，Zookeeper 将在后续版本中被完全移除。

## 33. MyBatis 与 Hibernate 有哪些不同？

🟡 **中等** | 标签：后端, MyBatis

MyBatis 与 Hibernate 都是优秀的持久层框架，但核心区别在于设计理念与灵活性。

Hibernate 是全自动 ORM 框架，致力于将对象与数据库表完全映射。开发者通过操作对象即可，框架自动生成并执行 SQL，提供高度抽象化。其优势是开发效率高，数据库无关性好，支持复杂的关联映射和完整的对象生命周期管理。但这也可能带来性能调优困难，生成的 SQL 不易优化。

MyBatis 是半自动 ORM 框架，核心是 SQL 与对象的映射关系。它不屏蔽 SQL，允许开发者直接编写、优化 SQL 语句，并通过 XML 或注解进行灵活配置。这种方式上手门槛稍高，需要开发者精通 SQL，但提供了对数据库操作的极大控制力，特别适合处理复杂查询、优化性能以及对接存量复杂数据库。

总结来说，Hibernate 适合追求开发效率、对象模型清晰、数据库相对简单的场景。MyBatis 则更适合需要精细 SQL 控制、高性能要求、以及复杂查询优化的项目。在实际技术选型中，需根据项目复杂度和团队技能权衡。

## 34. MySQL InnoDB 引擎中的聚簇索引和非聚簇索引有什么区别？

🟡 **中等** | 标签：后端, MySQL, 数据库

## 聚簇索引与非聚簇索引的区别

**聚簇索引（Clustered Index）**
- 数据和索引存储在一起，叶子节点直接存储**完整的数据行**
- InnoDB 中主键索引就是聚簇索引，每张表**只能有一个**
- 如果未定义主键，InnoDB 会选择唯一非空索引；若都没有，则隐式生成一个 6 字节的 RowID

**非聚簇索引（Secondary Index）**
- 即二级索引/辅助索引，叶子节点存储的是**主键值**而非完整数据
- 一张表可以有多个二级索引

**核心区别**

| 对比项 | 聚簇索引 | 非聚簇索引 |
|--------|----------|------------|
| 叶子节点内容 | 完整数据行 | 主键值 |
| 数量限制 | 每表仅一个 | 可多个 |
| 查询效率 | 直接获取数据，速度快 | 需要**回表**查询 |

**回表机制**：通过二级索引查找到主键后，需再回到聚簇索引中查找完整数据，这就是"回表"。使用**覆盖索引**（查询字段已在索引中）可以避免回表，提升性能。

**简单记忆**：聚簇索引像字典正文按拼音排序，数据本身有序；非聚簇索引像书的目录，指向数据所在位置。

## 35. MySQL 三层 B+ 树能存多少数据？

🟡 **中等** | 标签：后端, MySQL, 数据库

在MySQL InnoDB存储引擎中，一个三层B+树能存储的数据量主要取决于每个节点的大小（通常为16KB）、索引键的大小以及数据行的大小。以16KB页大小和bigint（8字节）主键为例进行估算：

1.  **根节点（第一层）**：作为索引页，可存储约 **1170** 个指针（`(16*1024) / (8+6) ≈ 1170`）。
2.  **非叶子节点（第二层）**：每个节点同样存储约1170个指针。因此，第二层可指向 **1170 * 1170 ≈ 1,368,900** 个叶子节点。
3.  **叶子节点（第三层）**：存储实际数据行。假设每行数据大小为1KB，则一个16KB的页可存储约 **16** 行数据。

**总计**：三层B+树可存储的数据行数约为 `1170 * 1170 * 16 ≈ 2190万` 行。

**核心要点**：
-   **行大小是关键**：若每行数据较小（如100字节），则存储行数可上亿；若行很大（如8KB），则只能存储约 **2000多万** 行。
-   **主键选择**：使用更小的自增主键（如4字节int）能显著增加每页的索引指针数量，从而提升总容量。
-   **实际场景**：该估算提供了一个理论参考值。实际数据库中，行大小不一、存在页分裂等因素会影响最终值

## 36. MySQL 中使用索引一定有效吗？如何排查索引效果？

🟡 **中等** | 标签：后端, MySQL, 数据库, 场景题

索引并非在所有场景下都有效。索引失效的常见情况包括：对索引列使用函数或表达式、隐式类型转换、使用LIKE左模糊查询、OR条件连接非索引列、违反最左前缀原则等。此外，当查询数据量超过表的一定比例时，优化器可能选择全表扫描。

排查索引效果主要通过以下方法：首先使用EXPLAIN分析SQL执行计划，重点查看type、key、rows、Extra字段，确认是否命中索引及扫描行数。其次开启慢查询日志，捕获执行时间超过阈值的SQL。还可以通过SHOW INDEX查看索引基数，判断区分度是否足够。对于复杂场景，可使用SHOW PROFILE分析SQL各阶段耗时。最后建议结合具体业务数据特点，定期审查和优化索引设计，避免冗余和不当索引。

## 37. MySQL 中如何解决深度分页的问题？

🟡 **中等** | 标签：后端, MySQL, 数据库, 场景题

深度分页是指使用 `LIMIT offset, size` 语句，当 `offset` 值非常大时，查询性能会急剧下降。因为MySQL需要先读取 `offset + size` 行，然后丢弃前 `offset` 行，当偏移量达到百万级时，扫描大量数据导致I/O与CPU开销巨大。

核心优化思路是避免使用大的 `offset`。主要方案有两种：

1.  **子查询优化**：先通过覆盖索引快速定位主键ID，再关联查询获取完整数据。例如：`SELECT * FROM table WHERE id >= (SELECT id FROM table ORDER BY id LIMIT 1000000, 1) LIMIT 10;`。此方法利用子查询直接在索引上完成偏移，将随机IO转变为了顺序IO。
2.  **游标分页（推荐）**：记录上一页最后一条记录的ID或唯一值，使用 `WHERE` 条件定位。例如：`SELECT * FROM table WHERE id > last_max_id ORDER BY id LIMIT 10;`。这种方式完全避免了 `OFFSET`，复杂度稳定，性能最优，但要求分页连续，不能跳页。

对于需跳页的场景，可考虑使用Elasticsearch等搜索引擎，或通过业务设计限制可跳页数。

## 38. MySQL 中如何进行 SQL 调优？

🟡 **中等** | 标签：后端, MySQL, 数据库, 场景题

SQL 调优核心是减少查询时间与资源消耗，可从以下几方面入手：

**1. 分析查询计划（EXPLAIN）**
使用 `EXPLAIN` 分析 SQL 执行计划，重点关注 `type`（避免全表扫描 ALL）、`key`（是否使用索引）、`rows`（扫描行数）和 `Extra`（如 Using filesort/Using temporary 需优化）。

**2. 索引优化**
- 为 `WHERE`、`JOIN`、`ORDER BY` 字段创建合适索引。
- 遵循最左前缀原则，避免索引失效（如对索引列使用函数、隐式类型转换、`LIKE` 以通配符开头）。
- 避免过度索引，减轻写操作负担。

**3. SQL 语句优化**
- 避免 `SELECT *`，只查询必要字段。
- 用 `JOIN` 替代子查询，或在适当场景使用临时表。
- 分页优化：避免大偏移量（`LIMIT 100000, 10`），改用记录上一页最大ID过滤。
- 避免在 `WHERE` 中对字段进行运算或函数操作。

**4. 表结构设计**
- 选择合适数据类型，优先用整型、短字符串。
- 适度范式化以减少冗余

## 39. MySQL 中如果发生死锁应该如何解决？

🟡 **中等** | 标签：后端, MySQL, 数据库

MySQL死锁是指两个或多个事务相互持有对方需要的锁，导致都无法继续执行。解决死锁需从预防、检测和处理三方面入手。

首先，数据库层面，InnoDB引擎默认开启死锁检测（innodb_deadlock_detect），会自动回滚代价较小的事务，破坏死锁环。但检测本身消耗性能，高并发下可能成为瓶颈。也可设置锁等待超时（innodb_lock_wait_timeout），让长时间等待的事务自动回滚。

其次，应用层优化是关键。一是保持事务短小精悍，快速完成并释放锁；二是优化SQL，为查询条件添加合适索引，避免全表扫描导致锁升级；三是规范访问顺序，确保所有事务以相同顺序访问资源（如先访问表A再访问表B），从根本上避免循环等待。

最后，事后需分析。通过`SHOW ENGINE INNODB STATUS`命令查看最近死锁日志，或查询`innodb_trx`等系统表，定位冲突SQL和事务，针对性优化。应用代码中应设计健壮的重试机制，当捕获死锁错误时，对回滚的事务进行合理重试。

## 40. MySQL 中有哪些锁类型？

🟡 **中等** | 标签：后端, MySQL, 数据库

MySQL 中的锁主要分为以下类型：

**1. 全局锁：** 对整个数据库实例加锁，主要用于全库逻辑备份，保证数据一致性，但会影响业务。

**2. 表级锁：** 锁定整张表，并发度低。
   - **表锁：** 手动加 `LOCK TABLES` 读锁或写锁。
   - **元数据锁（MDL）：** 自动加锁，当对表进行查询时加MDL读锁，修改表结构时加MDL写锁。
   - **意向锁：** InnoDB 自动添加，用于表明事务稍后对表中的行需要加哪种类型的锁（意向共享锁/意向排他锁），以协调行锁与表锁的关系。

**3. 行级锁：** 仅锁住需要的行，并发度高，是 InnoDB 的默认实现。
   - **记录锁：** 锁定索引中的一条记录。
   - **间隙锁：** 锁定索引记录之间的间隙，防止幻读。
   - **临键锁：** 记录锁 + 间隙锁，是 InnoDB 在可重复读隔离级别下锁的默认算法。

**4. 共享锁（S锁）与排他锁（X锁）：**
   - 共享锁：允许事务读一行数据。
   - 排他锁：允许事务删除或更新一行数据。

**5. 乐观锁与悲观锁：** 并非具体实现，而是并发控制思想。
   - 悲观锁：假设冲突会发生，如使用 `SELECT ... FOR UPDATE`。
   - 乐观锁：假设冲突少，通过版本号或时间戳在更新时检测冲突。

在实际应用中，需根据业务场景和并发需求选择合适的锁策略，行级锁能有效提高并发性能，但需注意避免死锁。

## 41. MySQL 中的回表是什么？

🟡 **中等** | 标签：后端, MySQL, 数据库

回表是MySQL中InnoDB存储引擎执行查询时的一种现象。当使用非聚簇索引（即二级索引）查找数据时，如果查询所需字段未包含在该索引中，存储引擎需要通过索引叶子节点中存储的主键值，再回到聚簇索引（主键索引）中查找完整的行数据，这个过程称为回表。

回表会导致额外的磁盘I/O和查询开销，影响性能。例如，在`user`表中，若`name`列有索引，执行`SELECT * FROM user WHERE name='Alice'`时，会先通过`name`索引找到主键ID，再通过主键索引回表获取所有字段。

优化回表的方法主要有：
1. **使用覆盖索引**：将查询所需字段都包含在索引中，使查询仅通过索引即可完成，避免回表。
2. **合理设计索引**：遵循最左前缀原则，避免创建冗余索引。
3. **利用索引下推（ICP）**：MySQL 5.6+允许在索引遍历时直接过滤部分条件，减少回表次数。

理解回表机制有助于我们优化SQL和索引设计，提升数据库查询效率。

## 42. MySQL 中的索引数量是否越多越好？为什么？

🟡 **中等** | 标签：后端, MySQL, 数据库

不是越多越好。索引能显著提升查询性能，但过多索引会带来负面影响：首先，每次对数据进行增、删、改操作，数据库都需要同步维护所有相关索引，这会大幅降低写操作性能，增加 I/O 负担。其次，索引本身也占用磁盘空间，过多索引会快速消耗存储资源。再者，优化器在复杂查询时需要评估和选择众多索引，可能增加查询计划生成的时间。

因此，应遵循“按需创建”原则，根据高频查询、排序、分组的字段来设计索引，优先使用覆盖索引和联合索引来优化查询。定期审查并删除无用或重复的索引，以平衡读写性能与资源消耗。

## 43. MySQL 中长事务可能会导致哪些问题？

🟡 **中等** | 标签：后端, MySQL, 数据库

MySQL长事务主要会导致以下问题：

**一、锁资源占用**
长事务持有行锁或表锁时间过长，导致其他事务阻塞等待，严重时引发死锁，显著降低系统并发能力。

**二、主从复制延迟**
长事务执行完成后才同步到从库，会导致主从数据长时间不一致，影响读写分离架构的数据准确性。

**三、回滚段膨胀**
事务执行期间产生的大量undo log需要保留，占用存储空间，增加清理负担，影响数据库整体性能。

**四、Buffer Pool污染**
长事务访问的数据页会常驻内存，可能将热点数据挤出缓存，降低缓存命中率，影响查询效率。

**五、连接资源耗尽**
长时间占用数据库连接，可能导致连接池被占满，新请求无法获取连接而失败。

**六、回滚风险**
长事务若执行失败，回滚代价极大，可能长时间阻塞数据库，甚至影响可用性。

**优化建议：**
- 将大事务拆分为多个小批次执行
- 避免在事务中进行耗时的网络调用或复杂计算
- 设置合理的锁等待超时时间
- 通过监控及时发现并处理长事务

## 45. MySQL 的乐观锁和悲观锁是什么？

🟡 **中等** | 标签：后端, MySQL, 数据库

MySQL的乐观锁和悲观锁是处理并发数据访问的两种策略。

**悲观锁** 假设冲突总会发生，因此在操作数据前先加锁，阻止其他事务修改。在MySQL中，通过`SELECT ... FOR UPDATE`或`SELECT ... LOCK IN SHARE MODE`在数据库层面实现。它依靠数据库的锁机制，适合写多、冲突频繁的场景，但可能影响并发性能。

**乐观锁** 假设冲突罕见，只在数据提交更新时检查是否被其他事务修改过。通常通过在数据表中增加一个**版本号字段**实现。读取数据时获取版本号，更新时携带该版本号执行`UPDATE ... SET version = version + 1 WHERE id = ? AND version = ?`。若更新成功，表示期间无冲突；若失败（更新行数为0），则表示数据已被修改，需重新处理。它避免了锁开销，适合读多写少、冲突较少的场景，但需要应用层实现冲突检测逻辑。

**核心区别**：悲观锁通过数据库锁阻塞并发，乐观锁通过版本比对实现乐观更新。选择取决于业务场景的冲突频率和性能要求。

## 46. MySQL 的存储引擎有哪些？它们之间有什么区别？

🟡 **中等** | 标签：后端, MySQL, 数据库

MySQL 支持多种存储引擎，每种引擎针对不同的应用场景进行了优化，可以根据需要选择。最核心、最常用的有以下两种：

1.  **InnoDB**：这是 MySQL 5.5 版本之后的**默认存储引擎**。它最大的特点是**支持事务处理（ACID）、行级锁和外键约束**。InnoDB 采用**聚簇索引**，将主键索引与数据存储在一起，因此对主键查询的性能极高。它通过多版本并发控制（MVCC）来实现高并发下的读一致性，非常适合于需要高可靠性、高并发读写以及事务支持的业务场景，如电子商务、金融系统等。

2.  **MyISAM**：这是 MySQL 5.5 之前的默认引擎。它的主要特点是**不支持事务，仅支持表级锁**，因此在写操作（特别是并发写）较多时容易产生锁竞争。但它的结构简单，**查询速度非常快**，支持全文索引，且在只读或读多写少的应用（如数据仓库、日志系统）中性能表现优异。数据以独立的文件形式存储，便于备份和迁移。

**其他常见引擎包括**：
*   **Memory**：数据存储在内存中，访问速度极快，但服务重启后数据会丢失。常用于临时表或缓存。
*   **Archive**：支持高并发的插入操作，主要用于日志和数据采集，不支持删除和修改。
*   **CSV**：数据以逗号分隔的文本文件存储，便于与其他系统交换数据。

**核心区别总结**：
*   **事务支持**：仅 InnoDB 支持。
*   **锁粒度**：InnoDB 支持行级锁（并发高）；MyISAM 仅支持表级锁。
*   **外键**：仅 InnoDB 完整支持。
*   **索引结构**：InnoDB 是聚簇索引；MyISAM 是非聚簇索引。
*   **数据存储**：InnoDB 数据和索引存储在表空间文件中；MyISAM 以独立文件存储索引和数据。

在现代应用中，**InnoDB 因其出色的可靠性和并发性能，已成为绝大多数场景的首选**。MyISAM 则因其简单和速度，在某些特定只读场景中仍有一席之地。

## 47. MySQL 的索引下推是什么？

🟡 **中等** | 标签：后端, MySQL, 数据库

**索引下推（Index Condition Pushdown, 简称ICP）** 是MySQL 5.6版本引入的一种查询优化策略，用于减少存储引擎访问基表的次数，从而提升查询性能。

其核心思想是：将原本在MySQL Server层进行的索引条件过滤操作，“下推”到存储引擎层完成。传统查询流程中，存储引擎根据索引找到记录后，需要回表将完整记录返回给Server层，再由Server层进行WHERE条件过滤。而启用ICP后，对于索引中包含的查询条件，存储引擎会在索引遍历过程中直接进行判断和过滤，只有满足索引条件的记录才会去回表取完整数据行，大大减少了不必要的回表和数据传输。

**工作原理与优势：**
当查询使用联合索引时，如果WHERE条件中包含索引列范围查询（如>`、`<`）后的等值查询条件，ICP会非常有效。例如，在联合索引`(a, b)`上执行`WHERE a > 1 AND b = 10`，在没有ICP时，索引只能利用`a`列进行范围扫描，所有`a > 1`的索引项都需要回表。启用ICP后，存储引擎在索引遍历时会同时检查`b = 10`这个条件，仅当两者都满足时才回表，显著减少了回表次数和Server层需要处理的数据量。

总结来说，索引下推通过将部分WHERE条件判断下沉到存储引擎层，优化了索引的使用效率，尤其适用于联合索引的场景，是MySQL查询优化中的一个重要特性。

## 48. MySQL 的覆盖索引是什么？

🟡 **中等** | 标签：后端, MySQL, 数据库

覆盖索引（Covering Index）是指一个索引包含了查询所需的全部字段，使得查询只需扫描索引而不需要回表查询数据行。  

**核心原理**：当使用覆盖索引时，MySQL 通过索引就能直接获取所需数据（如 SELECT 字段或 WHERE 条件字段都在索引中），无需再通过主键到聚簇索引中查找完整行数据，减少了磁盘 I/O 和随机读取。  

**优势**：  
1. **减少回表**：避免二次查询，显著提升查询效率。  
2. **降低 I/O 开销**：索引通常比数据表小，读取索引更快。  
3. **优化统计查询**：对 COUNT、MAX 等聚合查询尤其有效。  

**使用场景**：  
- 高频查询的字段可以组合成联合索引，实现覆盖。  
- 需要优化的慢查询可通过 EXPLAIN 确认是否使用覆盖索引（Extra 列显示 “Using index”）。  

**注意事项**：  
- 不宜过度创建宽索引，以免增加写操作成本和存储空间。  
- 需权衡查询需求与索引维护代价。  

覆盖索引是优化数据库查询的重要手段，合理使用可大幅提升性能。

## 49. Netty 如何解决 JDK NIO 中的空轮询 Bug？

## 50. Netty 性能为什么这么高？

🟡 **中等** | 标签：Netty, 后端

Netty 的高性能主要源于以下几个核心设计：

1.  **基于 NIO 的 I/O 模型**：Netty 采用非阻塞 I/O（NIO），使用 Selector 多路复用机制，使得一个线程可以高效管理大量网络连接，避免了传统 BIO 模型中每个连接占用一个线程的资源浪费。

2.  **EventLoop 线程模型**：Netty 抽象出 EventLoop（事件循环）组件，将 I/O 读写、事件处理、定时任务等绑定到固定的线程上执行。通过主从 Reactor 多线程模型（通常是一个 Boss 线程负责接收连接，多个 Worker 线程负责处理 I/O），实现了高效的并发处理，减少了线程上下文切换的开销。

3.  **高效的内存管理（ByteBuf）**：Netty 提供了优化的字节缓冲区 ByteBuf。它支持池化、使用读写索引分离避免了 NIO ByteBuffer 的 flip 操作，并内置了零拷贝技术（如 CompositeByteBuf、FileRegion），大大减少了内存拷贝和 GC 压力。

4.  **无锁化设计与串行化处理**：通过 EventLoop 的单线程执行关键 I/O 事件处理链，避免了复杂的同步锁竞争，提高了并发效率。对于非 I/O 任务，则可以异步提交给其他线程池处理，保证主流程的流畅。

5.  **丰富的性能优化细节**：包括针对 Linux 等系统的 epoll 空轮询 bug 的优雅解决策略、可配置的高性能编解码器、以及对序列化框架的深度集成等。

总结来说，Netty 通过优秀的 NIO 抽象、精心的线程模型、极致的内存管理以及众多底层优化，构建了一个高效、稳定且易于扩展的网络应用框架。

## 51. Netty 是如何解决粘包和拆包问题的？

🟡 **中等** | 标签：Netty, 后端

在TCP协议中，数据以字节流形式传输，没有消息边界，导致粘包（多个消息合并）和拆包（一个消息被拆分）问题。Netty通过提供一系列解码器来高效解决此问题。

Netty的核心方案是使用**解码器（Decoder）**在入站处理中拆分数据：
1. **固定长度解码器（FixedLengthFrameDecoder）**：按固定字节数分割消息，适用于长度固定的数据格式。
2. **行分隔符解码器（LineBasedFrameDecoder）**：根据换行符（如\n）分割，适合文本协议。
3. **分隔符解码器（DelimiterBasedFrameDecoder）**：支持自定义分隔符，灵活处理多种协议。
4. **长度字段解码器（LengthFieldBasedFrameDecoder）**：通过消息头中的长度字段读取数据，是最常用的方案，可处理变长消息。

这些解码器基于Netty的ChannelHandler机制，自动将字节流转换为完整的业务消息，简化开发。Netty还允许自定义解码器，确保高扩展性。通过这种设计，Netty有效避免了粘包拆包问题，提升了网络通信的可靠性。

## 52. Netty 采用了哪些设计模式？

🟡 **中等** | 标签：Netty, 后端

Netty 主要采用了以下设计模式：

1. **Reactor 模式**：Netty 基于 Reactor 模式构建网络框架。主从 Reactor 线程模型中，主 Reactor 负责 Accept 新连接并分发给从 Reactor，从 Reactor 负责处理连接上的 I/O 读写事件。该模式实现了高效的事件驱动与多线程并发处理。

2. **责任链模式**：通过 `ChannelPipeline` 和 `ChannelHandler` 实现。Pipeline 作为责任链容器，Handler 是链上的处理器。入站/出站事件沿 Pipeline 传递，每个 Handler 可选择处理或传递给下一个，实现了解耦和灵活的编排。

3. **工厂模式**：如 `EventLoopGroup` 和 `Channel` 的创建，通常由专门的工厂方法生成。`Bootstrap` 和 `ServerBootstrap` 作为引导类，封装了创建过程，隐藏了复杂实现细节。

4. **装饰器模式**：在包装已有组件时使用，例如对 `Channel` 进行功能增强，`SslHandler`、`LoggingHandler` 等通过装饰原生 Channel 添加附加功能。

5. **观察者模式/监听器模式**：Netty 中通过 `ChannelFuture` 和 `FutureListener` 实现。异步操作完成后通过回调通知监听者，实现非阻塞的事件通知机制。

这些模式共同支撑了 Netty 的高性能、可扩展性与易用性，是其设计的精髓所在。

## 53. OSI 七层模型是什么？

🟡 **中等** | 标签：网络

OSI七层模型是国际标准化组织（ISO）提出的网络通信框架，将网络通信过程划分为七个功能层，每层各司其职、协同工作，实现数据的可靠传输。

从底层到顶层依次为：
1. **物理层**：负责在物理媒介上传输原始比特流，涉及接口、电压、线缆等硬件标准。
2. **数据链路层**：将比特流封装成帧，提供节点到节点的可靠传输，并进行物理寻址（MAC地址）和错误检测。
3. **网络层**：负责逻辑寻址（IP地址）和路由选择，实现不同网络之间的数据转发。
4. **传输层**：提供端到端的可靠或不可靠数据传输（如TCP/UDP），进行流量控制和差错恢复。
5. **会话层**：建立、管理和终止应用程序之间的会话连接。
6. **表示层**：处理数据格式转换、加密解密和压缩解压缩，确保不同系统间的数据可读性。
7. **应用层**：直接为用户应用程序提供网络服务接口（如HTTP、FTP、SMTP协议）。

OSI模型通过分层解耦简化了网络设计，虽然实际中更多使用TCP/IP四层模型，但OSI仍是理解网络通信原理和协议栈的重要基础。

## 54. RabbitMQ 中无法路由的消息会去到哪里？

🟡 **中等** | 标签：RabbitMQ, 消息队列, 后端

在 RabbitMQ 中，无法路由的消息主要有两个去向，具体取决于生产者的设置和交换机的配置：

**1. 进入死信交换机（DLX）**：这是最常见且推荐的方式。当消息无法被任何队列路由时，如果消息或队列配置了 `dead-letter-exchange` 属性，该消息会被转发到指定的死信交换机。随后，可以将其路由到死信队列进行集中处理、记录或人工干预。这实现了问题的隔离和解耦。

**2. 被直接丢弃**：如果消息没有配置死信交换机，并且生产者发送消息时设置了 `mandatory` 参数为 `false`（默认值），那么无法路由的消息会被 RabbitMQ 服务端**静默丢弃**。这可能导致数据丢失，且难以排查。

**最佳实践**：为了避免消息丢失，生产者应设置 `mandatory=true`，并监听 `Return` 回调以感知无法路由的消息。同时，为关键业务配置完善的死信队列机制，确保所有“迷路”的消息都有迹可循、有处可去。

## 55. RabbitMQ 中消息什么时候会进入死信交换机？

🟡 **中等** | 标签：RabbitMQ, 消息队列, 后端

在RabbitMQ中，消息进入死信交换机（DLX）主要有以下四种情况：

1. **消息被消费者拒绝**：当消费者使用`basic.reject`或`basic.nack`拒绝消息，且设置了`requeue=false`（不重新入队）时，消息会进入DLX。
2. **消息TTL过期**：消息设置了过期时间（TTL），或者其所在队列设置了`x-message-ttl`参数，导致消息在队列中生存时间超限后被丢弃至DLX。
3. **队列达到最大长度**：当队列设置了`x-max-length`或`x-max-length-bytes`，且新消息入队时队列已满，则最早的消息会被丢弃至DLX。
4. **消息被丢弃**：在队列设置了`x-overflow`参数为`reject-publish`的情况下，若消息因上述第3条原因被丢弃，且生产者设置了`mandatory`标志，则消息可能通过`basic.return`退回；否则会直接进入DLX。

死信交换机常用于实现延迟队列、消息重试、异常监控等场景，是保障消息可靠性的重要机制。配置时需将死信交换机绑定到死信队列，并通过队列的`x-dead-letter-exchange`参数指定DLX。

## 56. RabbitMQ 怎么实现延迟队列？

🟡 **中等** | 标签：后端, 消息队列, RabbitMQ

RabbitMQ 实现延迟队列主要有两种方式：

**1. 利用 TTL + 死信交换机（经典方案）**
- **原理**：为消息或队列设置 TTL（存活时间），消息过期后成为死信，被转发到绑定的死信交换机，再路由到实际消费的队列，从而实现延迟。
- **实现步骤**：
  - 创建普通交换机和队列（TTL队列），并绑定；设置队列的 `x-dead-letter-exchange` 和 `x-dead-letter-routing-key` 属性，指向死信交换机和路由键。
  - 生产者将消息发送到 TTL 队列，消息在队列中存活指定时间后过期，自动转入死信队列。
  - 消费者监听死信队列，实现延迟消费。
- **注意**：若使用消息级别 TTL，需确保消息过期时不在队列头部（否则可能被立即丢弃），可通过设置队列的 `x-message-ttl` 统一控制。

**2. 使用延迟插件（rabbitmq_delayed_message_exchange）**
- **原理**：安装官方延迟插件，声明 `x-delayed-message` 类型的交换机，消息发送时在 Header 中设置延迟时间，交换机暂存消息，到期后自动投递。
- **实现步骤**：
  - 安装并启用插件。
  - 声明延迟交换机，类型为 `x-delayed-message`，并配置 `x-delayed-type`（如 direct）。
  - 生产者发送消息时，在 Header 中设置 `x-delay`（毫秒值），消息经交换机延迟后路由到队列。

**对比与选择**：TTL+死信方案无需额外组件，但灵活性较低（队列TTL统一，消息TTL需注意顺序）；延迟插件更直观且支持动态延迟时间，但需维护插件。实际中可根据需求和运维成本选择。

## 57. Redis 中如何保证缓存与数据库的数据一致性？

🟡 **中等** | 标签：后端, Redis

在Redis中保证缓存与数据库的数据一致性，常见方案有以下几种：

**1. 先更新数据库，再删除缓存**
- 优点：时序清晰，若缓存删除失败可重试。
- 缺点：在删除缓存前，其他请求可能读到旧数据。可通过**延迟双删**优化：在更新数据库后，延迟一段时间再次删除缓存，以覆盖并发读取的情况。

**2. 先删除缓存，再更新数据库**
- 缺点：并发下易出现不一致。例如：请求A删除缓存后，请求B读取旧数据并写入缓存，此时请求A才更新数据库，导致缓存中为脏数据。
- 解决方案：使用**延迟双删**或**分布式锁**保证操作串行化。

**3. 延迟双删**
- 核心流程：先删缓存 -> 更新数据库 -> 等待一个读请求完成周期 -> 再次删除缓存。
- 适用于对一致性要求较高且并发量大的场景。

**4. 基于消息队列的最终一致性**
- 数据库更新时发送消息到消息队列，消费者异步更新缓存。若失败则重试，保证最终一致。
- 结合**binlog订阅**（如Canal），实现更实时的缓存更新。

**5. 设置合理的缓存过期时间**
- 作为兜底方案，即使数据不一致，也能在过期后自动恢复。

**总结**：无银弹方案，需根据业务场景选择。高并发场景推荐“先更新数据库，再删缓存”结合延迟双删；对一致性要求极高可考虑分布式锁或强一致中间件；一般业务采用最终一致性方案即可。

## 58. Redis 中如何实现分布式锁？

🟡 **中等** | 标签：后端, Redis

Redis 实现分布式锁主要基于其单线程处理命令的特性以及原子操作。核心是使用 `SET` 命令的 `NX`（仅当键不存在时设置）和 `PX`（设置过期时间，单位毫秒）选项，以原子方式完成“设置键值”和“设置过期时间”两步，防止死锁。

基本命令为：`SET lock_key unique_value NX PX 30000`。其中 `unique_value`（如UUID）用于标识锁持有者，确保后续安全释放。

释放锁时，必须通过 **Lua脚本** 原子地执行“判断值匹配则删除”的操作，避免误删其他客户端持有的锁。脚本逻辑为：如果键存在且值匹配，则执行删除；否则返回0。

这种方法在主从架构下存在缺陷：主节点宕机可能导致锁丢失。为此，Redis官方提出了 **RedLock算法**，即依次向N个独立的Redis实例请求加锁，当多数节点（N/2+1）成功且耗时未超过锁的有效时间时，才认为加锁成功，从而提升可靠性。

**优点**：实现相对简单，性能高。**缺点**：需要处理好超时、续期（看门狗机制）以及节点故障场景下的可靠性问题。

## 59. Redis 中有哪些内存淘汰策略？

🟡 **中等** | 标签：后端, Redis

Redis 提供了 8 种内存淘汰策略，用于在内存达到上限时决定如何删除键。

策略主要分为两大类：
1.  **`volatile` 系列**：仅从设置了过期时间的键中淘汰。
    *   `volatile-lru`：使用 LRU（最近最少使用）算法淘汰。
    *   `volatile-lfu`：使用 LFU（最不经常使用）算法淘汰。
    *   `volatile-ttl`：淘汰即将过期的键（剩余存活时间最短）。
    *   `volatile-random`：随机淘汰。

2.  **`allkeys` 系列**：从所有键中淘汰。
    *   `allkeys-lru`：对所有键使用 LRU 算法。
    *   `allkeys-lfu`：对所有键使用 LFU 算法。
    *   `allkeys-random`：随机淘汰。

另有两种特殊策略：
*   **`noeviction`（默认策略）**：不淘汰任何键。当内存满时，写入操作会报错，读取操作正常。
*   **`volatile-lfu` 和 `allkeys-lfu`** 是 Redis 4.0 引入的更智能的策略，能更好地追踪访问频率。

**面试要点建议**：通常推荐使用 `allkeys-lru`（适用于缓存场景）或 `allkeys-lfu`（访问频率差异明显时）。选择 `volatile` 系列需确保键确实设置了过期时间。`noeviction` 适用于不允许数据丢失的持久化场景。

## 60. Redis 中的 Big Key 问题是什么？如何解决？

🟡 **中等** | 标签：后端, Redis

Redis 中的 Big Key 是指单个键（Key）存储的数据量过大，通常表现为：
- **String 类型**：值超过 10KB
- **Hash、List、Set、ZSet 类型**：元素数量超过 5000 个或占用内存过大。

**主要危害：**
1. **阻塞操作**：DEL 等命令可能耗时过长，阻塞其他请求。
2. **内存不均**：导致集群节点内存负载不均衡。
3. **网络拥塞**：大 Key 传输占用大量带宽。
4. **持久化风险**：可能引发 AOF 重写或 RDB 快照延迟。

**解决方案：**
1. **拆分 Key**：例如将大 Hash 拆分为多个子 Hash，使用 `hash_tag` 保证在同一 slot。
2. **压缩数据**：对 String 值进行压缩（如 gzip），或使用更高效的数据结构（如用 Bitmap 替代 Set）。
3. **惰性删除**：在业务低峰期异步删除大 Key（Redis 4.0+ 支持 `UNLINK` 命令）。
4. **监控与治理**：使用 `redis-cli --bigkeys` 扫描，定期清理或迁移。

## 61. Redis 中的缓存击穿、缓存穿透和缓存雪崩是什么？

🟡 **中等** | 标签：后端, Redis

**缓存击穿**：某个热点key突然失效，大量并发请求直接访问数据库。
- **解决方案**：1）使用互斥锁，确保重建缓存时只有一个线程访问数据库；2）设置热点数据永不过期。

**缓存穿透**：查询不存在的数据，缓存未命中，请求直接穿透到数据库。
- **解决方案**：1）布隆过滤器，拦截非法请求；2）缓存空值，并设置较短过期时间。

**缓存雪崩**：大量缓存key同时失效，或Redis服务宕机，导致请求涌向数据库。
- **解决方案**：1）随机化key的过期时间，避免集中失效；2）搭建Redis高可用集群，防止单点故障；3）服务降级与限流，保护数据库。

## 62. Redis 为什么这么快？

🟡 **中等** | 标签：后端, Redis

Redis 之所以快，主要基于以下几个核心设计：

**1. 基于内存，摒弃磁盘IO瓶颈**
Redis将数据存储在内存中，内存的读写速度比磁盘快数万倍，这是其高性能的根本。所有操作都在内存中完成，避免了传统数据库频繁的磁盘IO操作。

**2. 单线程模型，避免上下文切换**
Redis核心处理模块采用单线程。这避免了多线程编程中的锁竞争、线程上下文切换的开销，简化了数据结构实现，同时保证了命令的原子性。其高性能得益于高效的事件驱动模型和IO多路复用，而非多线程。

**3. IO多路复用，高效处理并发连接**
Redis使用epoll（Linux）等IO多路复用技术，单线程即可同时监听和处理大量客户端连接请求，实现了高并发下的高效网络通信。

**4. 精心优化的数据结构**
Redis内置了如简单动态字符串（SDS）、压缩列表、跳表等高效数据结构。针对不同的数据类型（如String、Hash、List、Set、ZSet）使用了最合适的底层结构，使得数据操作的时间复杂度极低。

**5. 通信协议高效**
客户端与服务端之间采用简单的RESP协议通信，解析简单，序列化开销小。

**总结来说**，Redis通过**内存存储**消除了最慢的IO环节，通过**单线程与IO多路复用**结合，在保证并发处理能力的同时避免了复杂性开销，再辅以**高效的数据结构**，共同造就了其卓越的读写性能。

## 63. Redis 主从复制的实现原理是什么？

🟡 **中等** | 标签：后端, Redis

Redis 主从复制基于**全量复制（初始化阶段）**与**增量复制（命令传播阶段）**机制实现数据同步。

1.  **连接与握手**：从节点（slave）向主节点（master）发送`PSYNC`命令，携带复制偏移量和运行ID，请求数据同步。
2.  **全量复制**：首次连接或无法增量复制时触发。主节点执行`BGSAVE`生成RDB快照，同时将后续写命令缓存到复制缓冲区。完成后，将RDB文件和缓冲区内的命令发送给从节点，从节点清空旧数据并加载新数据。
3.  **增量复制**：连接建立后，主节点将每个写命令通过复制缓冲区发送给从节点。从节点执行命令并确认，两者通过**复制偏移量**保持一致。若连接断开重连，从节点汇报偏移量，主节点仅同步缺失的数据（缓冲区中仍存在）。
4.  **心跳机制**：从节点每秒向主节点发送`REPLCONF ACK`上报偏移量，用于监控延迟和连接活性。

**核心**：通过RDB进行初始全量同步，之后利用内存缓冲区和偏移量实现高效增量同步，保证数据最终一致。主节点异步复制，因此存在短暂的数据不一致风险。

## 64. Redis 和 Memcached 有哪些区别？

🟡 **中等** | 标签：后端, Redis

Redis 与 Memcached 的核心区别如下：

1. **数据结构**：Redis 支持字符串、哈希、列表、集合、有序集合等多种复杂数据结构，支持原子操作；Memcached 仅支持简单的键值对（字符串）。

2. **持久化**：Redis 提供 RDB 和 AOF 两种持久化机制，可将数据保存到磁盘，重启后恢复；Memcached 数据仅存于内存，重启即丢失。

3. **集群与高可用**：Redis 原生支持主从复制、哨兵模式及 Cluster 分布式集群，具备高可用和水平扩展能力；Memcached 依赖客户端分片，无集群内数据同步机制。

4. **内存管理**：Redis 可设置过期策略，并通过淘汰机制（如 LRU）管理内存；Memcached 内存分配采用 Slab Allocator，可能出现内存碎片。

5. **性能与线程**：Memcached 多线程设计，对多核 CPU 利用率较高；Redis 单线程模型（6.0 后引入多线程 IO），避免线程竞争，数据操作原子性强。

6. **功能扩展**：Redis 支持发布订阅、Lua 脚本、事务等高级功能；Memcached 功能相对单一。

**适用场景**：Redis 适合需要复杂数据结构、持久化或高可靠性的场景；Memcached 更适合纯缓存、简单键值存取且对内存利用率要求高的场景。

## 65. Redis 在生成 RDB 文件时如何处理请求？

🟡 **中等** | 标签：后端, Redis

Redis在生成RDB快照时，采用**fork子进程**的方式来处理请求，以确保服务持续可用。

核心机制如下：
1.  **fork进程**：Redis主进程会调用`fork()`创建一个子进程来专门负责将数据写入RDB文件。
2.  **父子分工**：
    *   **父进程**：继续负责处理所有客户端的命令请求。对于读操作，直接处理。对于写操作，在修改数据前，系统会使用**Copy-On-Write（COW，写时复制）** 机制，将对应的数据页复制一份新的副本，在这个副本上进行修改，确保子进程看到的始终是`fork()`时那一刻的数据快照。
    *   **子进程**：独自将内存中的数据集（即`fork`时刻的快照）序列化写入磁盘上的RDB临时文件，完成后替换旧文件。
3.  **数据一致性**：整个过程无需阻塞服务，子进程读取的是不会变化的快照，而父进程在原始内存上通过COW机制处理写入，从而保证了在不中断服务的前提下，生成某一时刻的一致性数据快照。

这种方式最大限度地减少了持久化操作对正常服务的影响，但需注意，如果写请求非常频繁，COW可能导致内存占用暂时增加。

## 66. Redis 实现分布式锁时可能遇到的问题有哪些？

🟡 **中等** | 标签：后端, Redis

Redis分布式锁实现可能面临以下主要问题：

1.  **锁的过期与业务执行时间不匹配**：锁的生存时间（TTL）如果设置过短，业务未执行完锁便自动释放，会导致其他客户端获取锁，引发并发问题。设置过长则可能导致锁长期被占用。

2.  **可重入性**：简单的实现无法满足同一个客户端在持有锁期间再次加锁的需求，可能引发死锁。通常需要使用Hash结构存储重入次数来实现可重入锁。

3.  **Redis集群环境下锁的可靠性**：在主从架构下，当主节点宕机后，存储的锁信息可能未能及时同步到从节点，导致其他客户端在新主节点上重复加锁，锁的安全性被破坏。可考虑使用RedLock算法来提高可靠性。

4.  **业务阻塞导致锁提前释放**：客户端获取锁后，若发生长时间GC或网络延迟，锁可能因超时而自动释放。当客户端恢复执行后，它会误认为仍持有锁。解决方案是使用看门狗（Watchdog）机制，在锁持有期间定期续期。

## 67. Redis 性能瓶颈时如何处理？

🟡 **中等** | 标签：后端, Redis

当 Redis 出现性能瓶颈时，可按以下思路系统性排查与优化：

**1. 诊断定位瓶颈**
   - **慢查询分析**：通过 `SLOWLOG GET` 命令查看耗时操作，定位问题命令。
   - **监控指标**：观察 CPU、内存、网络IO 使用率，以及 `INFO` 命令中的命中率、连接数等。
   - **客户端分析**：检查是否存在网络延迟或客户端连接池配置不当。

**2. 针对性优化**
   - **优化数据结构与命令**：避免 `KEYS`、`HGETALL` 等O(N)复杂度命令，用 `SCAN` 分批迭代。对大 Key 进行拆分（如 Hash 分片）。
   - **减少大 Key 与热 Key**：通过 `redis-cli --bigkeys` 扫描大 Key，考虑压缩或删除；对热 Key 使用本地缓存或 Key 分散。
   - **优化持久化**：若使用 RDB，避免在高负载时执行 `BGSAVE`；若使用 AOF，选择合适的重写策略。
   - **合理配置**：调整 `maxmemory` 及淘汰策略，设置合理的超时时间，优化内核参数（如 `vm.overcommit_memory`）。

**3. 架构扩展**
   - **读写分离**：使用从库分担读流量。
   - **集群分片**：若单实例内存或并发瓶颈，使用 Redis Cluster 进行水平扩展。
   - **客户端优化**：使用连接池、Pipeline 批量操作，并对非核心业务降级或异步处理。

最终需结合具体场景（如数据量、QPS、读写比例）选择最合适的组合方案。

## 68. Redis 支持事务吗？如何实现？

🟡 **中等** | 标签：后端, Redis

Redis 支持事务。其实现主要基于以下机制：  
1. **事务保证**：通过 `MULTI`、`EXEC`、`DISCARD` 和 `WATCH` 命令实现。`MULTI` 标记事务开始，之后的命令会进入队列，`EXEC` 按顺序执行队列中的命令。  
2. **原子性**：事务中的命令要么全部执行，要么全部不执行（但**不支持回滚**）。若队列中命令有语法错误，所有命令会被取消；若运行时某条命令失败，其他命令仍会执行。  
3. **乐观锁机制**：`WATCH` 可监控键，若事务执行前键被修改，则事务会失败（CAS思想）。  
4. **非严格隔离性**：事务执行期间不会被其他命令打断，但未实现真正的隔离级别。  

**特点**：轻量级，性能高，但与传统数据库事务不同，更接近“批处理”。适合简单原子操作场景，复杂逻辑需结合 Lua 脚本增强。

## 69. Redis 数据过期后的删除策略是什么？

🟡 **中等** | 标签：后端, Redis

Redis 主要采用 **惰性删除** 与 **定期删除** 相结合的策略来处理过期键，并在内存不足时辅以 **内存淘汰策略**。

1.  **惰性删除**：当客户端访问一个键时，Redis 会检查其是否已过期。若过期，则删除并返回空值；否则正常返回。此策略对 CPU 友好，但可能占用过多内存。
2.  **定期删除**：Redis 会周期性地（默认每秒10次）随机抽取一批设置了过期时间的键进行检查，并删除其中已过期的键。通过调整执行频率和每次检查的键数量，它在 CPU 和内存消耗之间取得平衡。
3.  **内存淘汰策略**：当内存使用达到上限时，Redis 会根据配置的淘汰策略（如 allkeys-lru, volatile-lru 等）主动删除一部分键来腾出空间。这并非直接针对过期键，但能确保系统稳定运行。

**核心要点**：惰性删除保证访问时清理，定期删除主动清理一部分，两者配合减少内存浪费。内存淘汰是最后的保障机制，共同构成了 Redis 高效的内存管理策略。

## 70. Redis 的 Lua 脚本功能是什么？如何使用？

🟡 **中等** | 标签：后端, Redis

Redis的Lua脚本功能允许在服务端原子性地执行自定义逻辑。它通过将多个Redis命令和复杂操作封装为脚本，保证执行过程的原子性，避免并发竞争，并减少客户端与服务端的网络往返次数，从而提升性能。

**核心功能与优势**：
1.  **原子性执行**：脚本在Redis中单线程执行，中间不会被其他命令打断，确保操作完整性。
2.  **减少网络开销**：可将多个命令合并为一个脚本调用，降低网络延迟。
3.  **逻辑复用**：支持在脚本中实现条件判断、循环等复杂逻辑，实现如分布式锁、限流、库存扣减等场景。

**使用方法**：
- **执行脚本**：使用 `EVAL` 命令直接执行，或使用 `EVALSHA` 通过脚本SHA1校验和调用已缓存的脚本。
  ```redis
  EVAL "脚本内容" 键个数 [键...] [参数...]
  ```
- **Java中调用**：通过Jedis等客户端库，例如：
  ```java
  String script = "if redis.call('get', KEYS[1]) == ARGV[1] then return redis.call('del', KEYS[1]) else return 0 end";
  jedis.eval(script, Collections.singletonList(key), Collections.singletonList(value));
  ```

**典型场景**：
- 分布式锁：检查锁标识后释放，保证操作原子。
- 限流：通过键的计数和过期时间实现滑动窗口限流。
- 原子操作：如“检查并设置”（CAS），保证多步操作的一致性。

**注意**：脚本应尽量短小且无副作用，避免长时间阻塞Redis服务。

## 71. Redis 的 Pipeline 功能是什么？

🟡 **中等** | 标签：后端, Redis

Redis的Pipeline（管道）是一种将多个命令打包，然后一次性发送给服务端执行的技术，其主要目的是**减少网络往返次数（RTT），提升批量操作的执行效率**。

**核心原理**：客户端不立即等待每条命令的返回结果，而是将多条命令依次发送到缓冲队列，最后统一发送。服务端收到后会依次执行，并将所有结果打包返回。整个过程仅发生一次网络通信，避免了传统模式下每条命令都要单独等待请求响应的开销。

**主要优势**：
1.  **高性能**：特别适合需要批量执行多个独立命令的场景，如一次性读写多个键值，能显著降低延迟。
2.  **减少网络开销**：将多次IO操作合并为一次，有效节省网络资源。

**使用注意**：
*   Pipeline中的命令之间**没有依赖关系**，后续命令不依赖前一命令的结果。
*   它并非原子操作，执行期间可能会穿插其他客户端的命令。
*   与`MSET`、`MGET`等原生批量命令不同，Pipeline可以组合任意命令，灵活性更高。
*   需控制单次Pipeline的命令数量，避免一次性发送过多数据造成阻塞。

在Java中，Jedis、Lettuce等客户端库都提供了便捷的Pipeline API支持。

## 72. Redis 的 hash 是什么？

🟡 **中等** | 标签：后端, Redis

Redis 的 Hash 是一种键值对集合结构，专门用于存储对象类型的数据。它类似于 Java 中的 Map<String, Map<String, String>> 结构。

**核心特点**：
1.  **内部结构**：在元素较少时使用 ziplist（压缩列表）节省内存；当数据量增大时自动转换为 hashtable（哈希表）。
2.  **操作高效**：单个字段的读写时间复杂度为 O(1)，支持批量操作（如 `HMSET`、`HMGET`）。
3.  **节省内存**：相比为每个字段单独创建 key，Hash 能显著减少 key 的数量和内存开销。

**典型应用场景**：
- 存储用户信息（如 `user:1001` -> `{"name":"...", "age":"..."}`）。
- 存储商品信息、配置数据等结构化对象。
- 替代将对象序列化为 JSON 字符串的方式，支持灵活修改单个字段。

**简单示例**：
```
HSET user:1001 name "张三" age 30
HGET user:1001 name   # 返回 "张三"
```

总结：Redis Hash 是存储和管理对象的理想选择，兼具操作效率和内存优势。

## 73. Redis 的哨兵机制是什么？

🟡 **中等** | 标签：后端, Redis

Redis的哨兵（Sentinel）是一个用于实现Redis高可用性的分布式系统。它的主要作用是**监控主从Redis实例，并在主节点发生故障时，自动执行故障转移，将一个从节点提升为新的主节点**，并通知其他从节点和客户端更新连接。

其核心功能包括：

1.  **监控**：持续检查主节点和从节点是否正常工作。
2.  **通知**：当被监控的Redis实例出现问题时，可通过API通知管理员或其他应用程序。
3.  **自动故障转移**：当主节点不能正常工作时，哨兵会开始自动执行故障转移流程。它会选举出一个健康的从节点升级为新主节点，并让其他从节点复制这个新主节点。
4.  **配置提供者**：客户端在初始化时连接到哨兵，询问当前的主节点地址，从而始终能连接到正确的Redis主节点。

其工作原理基于一组哨兵进程，它们之间通过投票（类似Raft协议）来判断主节点是否下线（主观下线 -> 客观下线），并选举出一个“领导哨兵”来执行故障转移操作。

**需要注意的是**，哨兵本身是一套独立的分布式系统，需要合理部署（通常至少3个实例）以保证自身的高可用。对于开发者而言，通常只需在客户端配置中指向哨兵节点，即可由哨兵集群自动管理连接地址。

## 74. Redis 的持久化机制有哪些？

🟡 **中等** | 标签：后端, Redis

Redis 提供两种主要的持久化机制：**RDB（快照）** 和 **AOF（追加文件）**。

1.  **RDB**：在指定的时间间隔内，将内存中的数据集快照写入磁盘，生成一个压缩的二进制文件 `dump.rdb`。恢复时，直接加载该文件。其优点是文件紧凑、恢复速度快，适合备份和灾难恢复。缺点是可能丢失最后一次快照后的数据。
2.  **AOF**：以日志形式记录每一个写操作命令（追加到 `appendonly.aof` 文件末尾）。重启时，通过重放所有命令来重建数据。其优点是数据安全性更高（可通过配置`appendfsync`策略控制同步频率），文件可读。缺点是文件体积通常比RDB大，恢复速度较慢。
3.  **混合持久化**（Redis 4.0+）：结合了两者的优势。重写AOF时，将当前数据以RDB格式写入AOF头部，后续增量命令仍以AOF追加。这样既保证了重启时的加载速度，又减少了数据丢失风险。

生产环境通常建议开启**AOF**以保障数据安全，或根据业务需求选择**混合持久化**作为折中方案。

## 75. Redis 集群会出现脑裂问题吗？

🟡 **中等** | 标签：后端, Redis

Redis 集群确实可能出现脑裂问题。脑裂是指在网络分区故障下，集群被分割成多个独立的子集，每个子集都认为自己是主集群并选出各自的主节点，从而导致数据不一致。

其主要原因和典型场景是：部分节点之间网络中断，原先的主节点与多数派节点失联。如果少数派分区中的主节点仍在接受客户端写入，而多数派分区已选出新的主节点，那么在网络恢复后，原主节点的数据可能会被覆盖，造成数据丢失。

预防和解决脑裂的核心方案是**配置合理的预防策略和集群节点**：
1.  **设置 `min-replicas-to-write` (N) 和 `min-replicas-max-lag` (M) 参数**：要求主节点必须在至少有 N 个从节点连接且延迟不超过 M 秒的情况下才接受写入。这能确保少数派分区的主节点因不满足条件而停止服务，避免数据写入冲突。
2.  **优化网络架构**：确保节点间有稳定可靠的网络连接，避免单点故障。
3.  **依赖 Redis Cluster 内置机制**：集群使用 Gossip 协议通信和投票机制进行故障转移，但其本身无法完全杜绝网络分区下的数据不一致，因此仍需结合上述配置进行防护。

总结而言，通过合理配置“最少有效从节点”参数，可以大幅降低脑裂导致数据丢失的风险，是保障 Redis 集群数据一致性的关键实践。

## 76. Redis 集群的实现原理是什么？

🟡 **中等** | 标签：后端, Redis

Redis 集群通过数据分片与去中心化实现高可用与水平扩展。其核心原理如下：

1. **数据分片（哈希槽）**  
   集群将整个数据库划分为16384个槽位，每个节点负责一部分槽。数据通过CRC16哈希计算确定所属槽，再路由至对应节点，实现负载均衡。

2. **去中心化节点通信**  
   节点间通过Gossip协议交换状态信息（如槽分配、节点健康），维护集群元数据。客户端可连接任一节点，若请求数据不在当前节点，会返回重定向指令（MOVED）。

3. **故障转移与高可用**  
   主节点故障时，其从节点自动发起选举，接替主节点的槽位，保证服务连续性。集群通过投票机制确认节点失效，并触发故障转移。

4. **客户端支持**  
   客户端缓存槽位映射，直接路由请求。遇到节点变更时，通过重定向更新映射，减少跨节点跳转。

该设计在保证数据自动分片的同时，通过副本和故障转移机制实现了高可用性。

## 77. RocketMQ 的事务消息有什么缺点？你还了解过别的事务消息实现吗？

🟡 **中等** | 标签：消息队列, 后端, RocketMQ

RocketMQ 的事务消息通过**两阶段提交（半消息 + 二次确认）和事务状态回查机制**实现最终一致性，但其存在以下主要缺点：

1. **性能开销**：消息需要存储两次（半消息和正式消息），并触发回查，相比普通消息吞吐量降低，增加系统复杂度和延迟。
2. **回查机制局限性**：依赖业务实现回查接口，若业务未正确处理或应用宕机，可能导致消息长期挂起或状态不一致。回查频率和超时需要精细配置。
3. **数据一致性依赖应用**：仅保证消息与本地事务的最终一致，实际一致性取决于业务方对回查的实现质量，无法做到强一致性。
4. **功能限制**：早期版本不支持事务消息的定时和延时特性，且事务消息本身不支持消息过滤。

**其他事务消息实现**：
- **Kafka 的事务消息**：基于幂等生产者和事务协调器实现跨分区、跨主题的 Exactly-Once 语义，更适合流处理场景，但配置相对复杂。
- **本地消息表 + 消息队列**：通过数据库本地事务保障业务操作和消息记录的原子性，再由定时任务投递消息。实现简单、可靠性高，但侵入性强，需要维护消息状态表。

选择时需权衡一致性要求、性能成本及技术栈复杂度，RocketMQ 事务消息适合对最终一致性容忍度较高的业务场景。

## 78. Select、Poll、Epoll 之间有什么区别？

🟡 **中等** | 标签：操作系统

Select、Poll、Epoll 都是 I/O 多路复用技术，用于单个线程监控多个文件描述符（如 Socket 连接）的可读/可写状态，以提高服务器并发处理能力。它们的主要区别如下：

**1. 数据结构与连接数限制**
*   **Select**：使用位图（fd_set）管理文件描述符，受限于 `FD_SETSIZE`（通常为1024），连接数有上限。
*   **Poll**：使用链表结构存储 pollfd，理论上无固定连接数限制。
*   **Epoll**：使用红黑树和就绪链表管理文件描述符，连接数上限通常与系统内存相关，可支持海量连接。

**2. 性能与效率**
*   **Select/Poll**：每次调用都需将整个fd集合从用户态拷贝到内核态，并在内核中进行线性扫描（O(n)），当连接数增大时性能急剧下降。
*   **Epoll**：通过 `epoll_create` 在内核建立红黑树，通过 `epoll_ctl` 增删改，只在注册时拷贝一次。内核通过回调机制（网卡驱动注册中断处理函数）将就绪fd加入就绪链表，`epoll_wait` 只需检查链表（O(1)），效率高，不随连接数增长而显著下降。

**3. 触发模式**
*   **Select/Poll**：仅支持水平触发（LT），只要fd处于就绪状态，就会不断通知。
*   **Epoll**：支持水平触发（LT）和边缘触发（ET）。ET模式仅在状态变化（如新数据到达）时通知一次，效率更高，但编程需确保一次性处理完数据。

**总结**：Select/Poll适用于连接数少且活跃度高的场景；Epoll专为高并发、连接数多但活跃度不一定高的场景设计（如C10K问题），是Linux下高性能网络编程的首选。

## 79. Spring AOP默认用的是什么动态代理，两者的区别？

🟡 **中等** | 标签：后端, Spring

## Spring AOP 动态代理机制

Spring AOP 默认使用两种动态代理方式：**JDK 动态代理**和 **CGLIB 动态代理**。

### 两种代理方式

| 特性 | JDK 动态代理 | CGLIB 动态代理 |
|------|-------------|----------------|
| 实现原理 | 基于接口，通过 `java.lang.reflect.Proxy` 反射实现 | 基于继承，通过生成目标类的子类实现 |
| 要求 | 目标类**必须实现接口** | 目标类**不能是 final** |
| 性能 | 反射调用，创建快、调用慢 | 字节码生成，创建慢、调用快 |

### 默认策略

- **Spring 5 之前**：若目标类实现了接口，默认使用 JDK 动态代理；否则使用 CGLIB
- **Spring Boot 2.x 之后**：默认使用 **CGLIB 代理**，无需接口也可代理

可通过配置强制指定：
```properties
spring.aop.proxy-target-class=true   # 强制CGLIB
spring.aop.proxy-target-class=false  # 优先JDK
```

### 如何选择

- 对外提供的 SDK/框架，建议**面向接口编程 + JDK 代理**，解耦更彻底
- 内部业务代码，使用 **CGLIB 更方便**，不必强制实现接口
- 若方法被 `final` 修饰或类被 `final` 修饰，两种方式都无法代理

## 80. Spring Boot  的核心特性有哪些？

🟡 **中等** | 标签：Spring Boot, 后端

Spring Boot 的核心特性旨在简化 Spring 应用的开发与部署，主要包含以下几点：

1. **自动配置**：根据项目引入的依赖（Starter），Spring Boot 会自动推断并配置所需的 Bean，极大地减少了手动配置工作。
2. **起步依赖**：通过一系列预定义的、以`spring-boot-starter`为名的依赖描述符，开发者可以方便地引入一组相关技术栈（如Web、JPA、Test），自动管理依赖版本，避免冲突。

4. **Actuator 监控**：提供生产级特性，如健康检查、指标收集、环境信息等端点，便于对应用进行监控和管理。
5. **无需XML配置**：基于条件注解和约定优于配置的理念，全面拥抱 Java 配置和自动配置，基本消除了XML配置文件。
6. **开箱即用**：旨在快速启动和运行新项目，提供默认配置，开发者可专注于业务逻辑。

总之，Spring Boot 通过约定、自动化与内嵌容器，显著提升了 Spring 应用的开发效率和部署便捷性。

## 81. Spring Boot 是如何通过 main 方法启动 web 项目的？

🟡 **中等** | 标签：Spring Boot, 后端

Spring Boot 通过 `main` 方法启动 Web 项目的流程如下：

首先，在启动类上使用 `@SpringBootApplication` 注解，它是一个复合注解，核心是 `@EnableAutoConfiguration`，用于启用自动配置机制。

调用 `SpringApplication.run()` 方法时，主要经历两个阶段：

    *   执行所有监听器的 `started` 事件。
    *   调用所有 `CommandLineRunner` 和 `ApplicationRunner` 的 `run` 方法。

## 82. Spring MVC 具体的工作原理？

🟡 **中等** | 标签：后端, Spring

Spring MVC 的核心工作原理是围绕前端控制器 DispatcherServlet 展开的，遵循“请求-处理-响应”的模型。具体流程如下：

1. **请求到达**：客户端请求首先被前端控制器 **DispatcherServlet** 接收。
2. **查找处理器**：DispatcherServlet 调用 **HandlerMapping**，根据请求 URL 找到对应的 **Controller**（处理器）。
3. **执行处理**：通过 **HandlerAdapter** 调用 Controller 的具体方法，执行业务逻辑，该过程通常涉及参数绑定、校验等。
4. **返回模型视图**：Controller 方法返回一个 **ModelAndView** 对象，其中 Model 包含数据，View 指定视图名称。
5. **解析视图**：DispatcherServlet 将视图名称传给 **ViewResolver**，解析为具体的 **View** 实例。
6. **渲染与响应**：View 使用 Model 中的数据进行渲染（如填充 JSP 模型），最终 DispatcherServlet 将渲染结果返回给客户端。

整个过程体现了 MVC 的分离：Controller 处理业务，Model 承载数据，View 负责展示。DispatcherServlet 作为协调中心，实现了各组件的松耦合，便于开发和扩展。

## 83. Spring 中的 DI 是什么？

🟡 **中等** | 标签：后端, Spring

Spring中的DI（Dependency Injection，依赖注入）是Spring框架的核心机制之一。它指对象所需的依赖不由自身创建，而由外部容器（如Spring IoC容器）在运行时动态注入。其核心思想是“控制反转”，将对象的创建和依赖管理交给容器，降低代码耦合度。

DI主要有两种实现方式：构造器注入和Setter注入。构造器注入通过构造函数传递依赖，保证依赖不可变；Setter注入通过setter方法设置依赖，灵活性更高。此外，Spring还支持字段注入（如使用@Autowired），但官方更推荐前两种方式。

DI的优势在于：提高代码可测试性（便于Mock依赖）、增强模块化（组件间松耦合）、便于维护和扩展。例如，一个Service类依赖Repository，通过DI可在不修改Service代码的情况下切换不同的Repository实现。

在Spring中，容器负责管理Bean的生命周期和依赖关系，开发者只需通过注解（如@Component、@Autowired）或XML配置声明依赖关系，容器会自动完成注入。这使得应用架构更清晰、更灵活。

## 84. Spring 如何解决循环依赖？

🟡 **中等** | 标签：后端, Spring

Spring 主要通过**三级缓存机制**和**提前暴露对象引用**来解决单例 Bean 的循环依赖问题，其核心流程如下：

1.  **三级缓存定义**：
    *   `singletonObjects`（一级缓存）：存放完全初始化好的 Bean。
    *   `earlySingletonObjects`（二级缓存）：存放早期曝光的对象（已实例化，但未填充属性）。
    *   `singletonFactories`（三级缓存）：存放 Bean 的对象工厂（用于生成早期引用或代理对象）。

2.  **解决流程**（以 A 依赖 B，B 依赖 A 为例）：
    *   创建 Bean A 时，先实例化 A，并将其对象工厂放入三级缓存。
    *   填充 A 的属性时发现依赖 B，转去创建 B。
    *   创建 B 并实例化后，在填充属性时发现依赖 A。此时，容器会从三级缓存中获取 A 的对象工厂，调用其方法得到 A 的早期引用（可能是原始对象或代理对象），并将此引用放入二级缓存（同时移除三级缓存条目）。
    *   B 完成属性填充和初始化，被放入一级缓存。
    *   回到 A 的创建流程，A 获得已初始化的 B 并完成填充和初始化，最终放入一级缓存。

3.  **重要前提**：
    *   循环依赖只对**单例作用域**的 Bean 有效。
    *   依赖注入方式必须是**属性注入或Setter注入**，**构造器注入**会导致循环依赖失败，因为必须在实例化时就获得依赖，此时对象工厂还未创建。

**总结**：Spring 通过将 Bean 的创建过程与属性赋值过程分离，并利用缓存暂时存储早期引用来打破循环，从而解决了大部分场景下的循环依赖问题。

## 85. Spring 有哪几种事务传播行为?

🟡 **中等** | 标签：后端, Spring

Spring框架定义了7种事务传播行为，用于控制事务方法嵌套调用时的事务边界：

**一、支持当前事务（4种）**
1. **REQUIRED**（默认）：加入当前事务，若无则新建。
2. **SUPPORTS**：有事务则加入，无则以非事务执行。
3. **MANDATORY**：必须在事务中调用，否则抛异常。
4. **REQUIRES_NEW**：新建独立事务，挂起当前事务（若有）。

**二、不支持当前事务（2种）**
1. **NOT_SUPPORTED**：以非事务执行，挂起当前事务（若有）。
2. **NEVER**：以非事务执行，若存在当前事务则抛异常。

**三、嵌套事务**
1. **NESTED**：在当前事务中创建保存点，实现嵌套回滚（外层提交则内层一起提交）。

**总结表格：**

| 传播行为 | 当前有事务 | 当前无事务 |
| :--- | :--- | :--- |
| **REQUIRED** | 加入 | 新建 |
| **REQUIRES_NEW** | 新建（挂起原事务） | 新建 |
| **SUPPORTS** | 加入 | 非事务执行 |
| **NOT_SUPPORTED** | 非事务执行（挂起原事务） | 非事务执行 |
| **MANDATORY** | 加入 | 抛异常 |
| **NEVER** | 抛异常 | 非事务执行 |
| **NESTED** | 嵌套事务（保存点） | 新建 |

**核心考点：**
- **REQUIRED** 与 **REQUIRES_NEW** 的区别：前者共享事务上下文，后者完全独立。
- **NESTED** 与 **REQUIRES_NEW** 的区别：NESTED的回滚不影响外层，REQUIRES_NEW的提交/回滚完全独立。
- **MANDATORY** 用于强制调用方必须管理事务。

实际应用中，**REQUIRED** 是最常用的选择，适合大多数业务方法。选择传播行为时需结合业务逻辑的数据一致性要求。

## 86. SpringBoot 是如何实现自动配置的？

🟡 **中等** | 标签：Java, Spring Boot, 后端

SpringBoot 实现自动配置的核心在于 `@EnableAutoConfiguration` 注解（通常由 `@SpringBootApplication` 复合注解包含）。其流程可以概括为：

1. **触发**：当应用启动时，`@EnableAutoConfiguration` 会触发自动配置机制。
2. **加载候选组件**：SpringFactoriesLoader 会从所有引入的 JAR 包的 `META-INF/spring.factories` 文件中，读取 `org.springframework.boot.autoconfigure.EnableAutoConfiguration` 键下配置的**自动配置类全限定名列表**。
3. **条件装配**：这些配置类都使用了 `@Conditional` 系列注解（如 `@ConditionalOnClass`、`@ConditionalOnMissingBean`）。Spring 会根据当前应用的类路径、已有 Bean 定义等环境条件，**判断是否需要应用该配置**。例如，只有当类路径下存在 `DataSource.class` 时，数据源的自动配置类才会生效。
4. **配置生效**：满足条件的配置类会将其中的 `@Bean` 方法定义加载到 Spring 的应用上下文中，从而自动完成组件的创建和配置，如数据源、Web 服务器、模板引擎等。

简而言之，SpringBoot 通过这种 **“约定优于配置”** 的思想，根据项目引入的依赖和环境，智能地决定并加载默认配置，极大地简化了项目的初始搭建和开发过程。开发者可以通过自定义 `application.properties` 文件或 `@Configuration` 类来覆盖这些默认值。

## 87. TCP 滑动窗口的作用是什么？

🟡 **中等** | 标签：网络

TCP滑动窗口是传输层实现流量控制和保证可靠传输的核心机制。其主要作用体现在以下三点：

第一，**实现流量控制**。接收方通过TCP报文中的窗口大小字段，动态告知发送方自己当前的接收缓冲区剩余空间。发送方根据这个“窗口”值调整发送速率，确保不会因发送过快而导致接收方数据溢出，从而避免丢包和资源浪费。

第二，**保证传输可靠性**。它允许发送方在未收到确认的情况下，连续发送窗口内的多个数据段。通过序列号和确认应答机制，接收方可以对数据进行有序重组和校验。若发生丢包，发送方会根据确认信息对窗口内未被确认的数据进行重传，确保数据完整送达。

第三，**提升传输效率**。滑动窗口机制将单个数据包的逐次应答变为批量应答，显著减少了网络等待时间，提高了链路利用率。窗口大小可根据网络状况动态调整，是实现高效、可靠数据传输的关键设计。

## 88. TCP 的粘包和拆包能说说吗？

🟡 **中等** | 标签：网络

TCP的粘包和拆包是**面向字节流**的传输特性导致的问题。由于TCP没有消息边界，发送方多次写入的数据可能被接收方一次性读取（粘包），或一次写入的数据被分次读取（拆包）。

**产生原因主要有两点：**
1.  **发送端优化**：Nagle算法等机制可能将多个小包合并发送。
2.  **接收端读取时机不确定**：应用层读取缓冲区数据的时机和大小无法预知。

**核心解决方法在应用层协议设计：**
1.  **定长消息**：规定每条消息固定长度，不足则补位。
2.  **分隔符**：使用特殊字符（如`\n`）标识消息边界。
3.  **长度字段**：在消息头部包含消息体的长度。这是最常用且可靠的方式（如HTTP的`Content-Length`头）。接收方先解析长度，再读取对应字节的数据。

例如，采用“头部长度+消息体”协议，就能明确区分每条消息，有效解决粘包和拆包问题。

## 89. 为什么 MySQL 选择使用 B+ 树作为索引结构？

🟡 **中等** | 标签：后端, MySQL, 数据库

MySQL 选择 B+ 树作为索引结构，主要基于以下核心优势：

1.  **磁盘 I/O 效率高**：B+ 树的**非叶子节点仅存储键值（索引）**，不存储数据。这使得每个节点（即磁盘的一个“页”）可以容纳更多的索引项，从而**大幅降低树的高度**。查询数据时，需要的磁盘 I/O 次数更少，性能更优。

2.  **范围查询与排序性能卓越**：B+ 树的所有**数据记录都存储在叶子节点**，且叶子节点之间通过**双向链表**相连。这使得进行范围查询（如 `BETWEEN`, `>`, `<`）和排序（`ORDER BY`）时，可以高效地顺序遍历叶子节点，而无需像 B 树那样进行中序遍历。

3.  **查询性能稳定**：由于数据只存在于叶子节点，任何查询都必须从根节点走到叶子节点，因此**查询的时间复杂度稳定为 O(log n)**。而 B 树的数据可能存在于非叶子节点，查询性能不稳定。

4.  **全面的索引能力**：与仅支持等值查询的哈希索引不同，B+ 树索引能高效支持**等值查询、范围查询、排序和最左前缀匹配**，是数据库中最通用和可靠的索引结构。

综上，B+ 树通过优化磁盘访问模式、支持高效范围操作并提供稳定查询性能，完美契合了数据库系统对索引的需求。

## 90. 为什么 Redis 设计为单线程？6.0 版本为何引入多线程？

🟡 **中等** | 标签：后端, Redis

Redis 初期设计为单线程，核心是为了利用内存操作的极致速度和简化设计。在纯内存操作的场景下，CPU 通常不是瓶颈，真正的瓶颈在于网络IO。单线程避免了多线程编程中的锁竞争和上下文切换开销，反而可能提升性能。其单线程配合IO多路复用机制（如epoll），足以高效处理大量并发连接，且保证了命令执行的原子性，简化了数据结构实现和持久化（如RDB）的复杂度。

Redis 6.0 引入多线程，主要是为了突破**网络IO瓶颈**。随着硬件性能提升和业务规模扩大，单线程处理网络读写（解析请求、写回响应）在高并发下成为性能天花板。新版本将网络数据读写、命令解析等操作交给多个IO线程并行处理，但**命令的执行仍然是单线程**，从而在保持核心操作简洁高效的同时，提升了网络吞吐能力。这解决了实际瓶颈，是对单线程模型的合理优化而非颠覆。

## 91. 为什么 RocketMQ 不使用 Zookeeper 作为注册中心呢？而选择自己实现 NameServer？

🟡 **中等** | 标签：RocketMQ, 消息队列, 后端

RocketMQ 选择自研 NameServer 而非使用 Zookeeper 作为注册中心，主要基于以下核心考量：

首先，**性能与效率优先**。Zookeeper 基于 ZAB 协议，强调强一致性，但写操作需多数节点确认，在高并发场景下可能成为性能瓶颈。而 NameServer 设计为轻量级无状态节点，专注服务发现与路由管理，支持快速读写，更适合消息队列对高吞吐、低延迟的需求。

其次，**简化架构与运维**。Zookeeper 集群需维护复杂状态，增加部署和故障排查难度。NameServer 无状态设计，节点独立，易于水平扩展，且故障恢复快，降低了系统运维成本。

再者，**减少外部依赖**。自研 NameServer 可深度集成 RocketMQ 逻辑，避免引入第三方组件，提升系统自主可控性，确保与消息队列核心功能的紧密协作。

最后，**业务场景适配**。RocketMQ 聚焦消息通信，NameServer 仅处理 Broker 和路由信息，功能精简；而 Zookeeper 通用性强但可能引入冗余机制，增加不必要的开销。

总之，NameServer 以简洁、高效的设计契合了 RocketMQ 对性能、稳定性和可维护性的要求，体现了“适合的才是最好的”架构理念。

## 92. 为什么 Spring 循环依赖需要三级缓存，二级不够吗？

🟡 **中等** | 标签：后端, Spring

Spring 中的循环依赖指的是两个或多个 Bean 相互引用，导致依赖链形成闭环。为了解决这个问题，Spring 采用了三级缓存机制。

三级缓存包括：一级缓存存放完全初始化的单例 Bean；二级缓存存放早期暴露的 Bean（可能是未完全初始化的对象或代理）；三级缓存存放 Bean 的工厂对象，用于生成早期引用。当出现循环依赖时，Spring 首先从三级缓存中获取工厂，创建并暴露早期引用，放入二级缓存，从而允许其他 Bean 引用这个未完成的 Bean，解决循环问题。

为什么二级缓存不够？因为如果只有二级缓存，对于需要 AOP 代理或延迟初始化的 Bean，无法在早期阶段正确生成代理对象。三级缓存通过工厂机制，允许在 Bean 初始化前动态创建代理，确保依赖注入的正确性。例如，当 Bean A 依赖 Bean B，而 B 又依赖 A 时，三级缓存能提前暴露 A 的代理，避免死锁。如果只用二级缓存，可能无法处理代理场景，导致循环依赖失败或异常。

总之，三级缓存提供了更灵活的解决方案，确保 Spring 在复杂依赖场景下的稳定性。

## 93. 为什么 TCP 挥手需要有 TIME_WAIT 状态?

🟡 **中等** | 标签：网络

TIME_WAIT状态存在于主动关闭方发送最后一个ACK后，等待2MSL（最大报文生存时间）才转为CLOSED。它主要解决三个问题：

1. **确保最终ACK可靠到达**：若ACK丢失，被动关闭方会重传FIN。TIME_WAIT状态使主动方仍能正确接收并重发ACK，保证双方正常关闭。

2. **防止旧连接报文干扰新连接**：网络中延迟的报文可能属于已关闭的连接。等待2MSL（通常60秒）可确保这些报文自然消散，避免被新连接误收。

3. **保证双向连接完全终止**：通过等待，确保双方所有报文均已失效，避免半关闭状态残留。

实际开发中，高并发服务器可能因大量TIME_WAIT导致端口耗尽。可通过调整内核参数（如tcp_tw_reuse、tcp_tw_recycle）或优化应用设计（如减少短连接、使用连接池）缓解，但需谨慎评估其对协议正确性的影响。

## 94. 为什么不选择使用原生的 NIO 而选择使用 Netty 呢？

🟡 **中等** | 标签：Netty, 后端

选择Netty而非原生NIO，主要基于以下几点考量：

**1. 开发复杂度与易用性**
原生NIO的API相对底层且复杂，需手动管理`Selector`、`Channel`、`Buffer`的交互，并处理多线程、事件循环和半包粘包等网络编程的固有难题。Netty对此进行了高度抽象，提供了清晰的事件驱动模型、`ChannelPipeline`与`ChannelHandler`责任链机制，开发者只需关注业务逻辑，极大提升了开发效率与代码可维护性。

**2. 性能与稳定性**
Netty在性能上经过极致优化，如使用内存池减少GC压力、支持零拷贝（通过`CompositeByteBuf`等）、针对Linux的`epoll` ET模式优化等。其内部线程模型（主从Reactor）能高效处理高并发连接。同时，它内置了完善的空轮询（epoll bug）解决方案和断线重连、心跳机制等，稳定性远超自行在原生NIO上构建的实现。

**3. 生态与行业验证**
Netty是众多知名项目（如Dubbo、Elasticsearch、RocketMQ）的底层网络框架，经受了海量生产环境的考验。它提供了丰富的协议编解码器（如HTTP/2、Protobuf）、工具类和社区支持，而使用原生NIO则需要自行实现或整合这些能力，成本高昂且易出错。

综上所述，Netty在**提升开发效率、保障高性能与高稳定性、拥有成熟生态**方面优势显著，是生产级网络应用的更优选择。

## 95. 为什么需要消息队列？

🟡 **中等** | 标签：消息队列, 后端

消息队列的核心作用主要体现在以下几个方面：

**1. 异步处理，提升响应速度**
生产者将消息发送到队列后即可返回，无需等待消费者处理完成。例如用户下单后，订单服务只需发送消息，后续的库存扣减、积分发放等操作异步完成，用户无需长时间等待。

**2. 流量削峰，保护后端服务**
在秒杀、促销等高并发场景下，大量请求直接打到数据库会导致崩溃。消息队列作为缓冲层，将瞬时高流量转化为平稳的消息流，让消费者按能力匀速处理。

**3. 系统解耦，提高扩展性**
生产者和消费者无需感知对方的存在，只需约定消息格式。当新增下游系统时，只需订阅消息队列即可，无需修改上游代码，降低了系统间的耦合度。

**4. 消息广播，一对多分发**
一条消息可以被多个消费者同时消费，实现广播能力，适用于日志收集、数据同步等场景。

**5. 提高系统可靠性**
消息持久化存储在队列中，即使消费者宕机，恢复后仍可继续消费，避免消息丢失。

常用的消息队列有RabbitMQ、RocketMQ、Kafka，各有侧重。选择时需综合考虑吞吐量、延迟、可靠性等需求。

当然，引入MQ也会带来消息一致性、消息丢失、顺序性等挑战，需要根据业务场景权衡使用。

## 96. 什么情况下需要使用分布式事务，有哪些方案？

🟡 **中等** | 标签：Spring Cloud, 微服务, 分布式事务, 后端

分布式事务主要在微服务架构中，当一个业务操作需要跨多个服务、且这些服务各自拥有独立数据库时使用。此时，传统的单机数据库事务无法保证跨服务的数据一致性。典型场景包括：1）跨服务调用，如电商下单涉及订单、库存、积分等不同服务；2）基于消息的异步操作，如支付成功后需异步更新多个下游系统；3）涉及多数据源的复杂业务，如新老系统数据同步。

主流解决方案包括：1）**2PC（两阶段提交）**：通过协调者统一管理参与者事务的提交或回滚，实现强一致性，但性能较低且存在协调者单点问题。2）**TCC（Try-Confirm-Cancel）**：将业务拆分为预留资源、确认执行、取消执行三个阶段，业务侵入性强但灵活性高。3）**可靠消息最终一致性**：利用消息队列（如RocketMQ事务消息）确保本地事务与消息发送的原子性，实现最终一致性，适用于异步场景。4）**Saga模式**：将长事务拆分为多个本地事务，每个事务都有对应的补偿操作，通过事件驱动实现正向与逆向流程，适合长流程业务。

在微服务中，常用Seata等开源框架简化实现。选择方案时需权衡一致性要求、性能影响及业务复杂度。

## 97. 什么是 AOP？

🟡 **中等** | 标签：后端, Spring

AOP（面向切面编程）是一种编程范式，旨在将横切关注点（如日志、事务、权限校验等）从业务逻辑中分离出来，实现关注点的模块化。

**解决的问题**：传统OOP中，日志记录、事务管理等通用功能会散布在多个业务方法中，导致代码重复、耦合度高、难以维护。AOP通过将这些功能抽取为独立的"切面"，实现了与业务逻辑的解耦。

**核心概念**：
- **切面（Aspect）**：封装横切关注点的模块，如日志切面
- **连接点（JoinPoint）**：程序执行的特定点，如方法调用
- **切入点（Pointcut）**：匹配连接点的表达式，定义哪些方法会被增强
- **通知（Advice）**：切面在连接点执行的动作，包括前置、后置、环绕、异常、最终通知五种类型

**实现原理**：Spring AOP基于动态代理实现。对于接口使用JDK动态代理，对于类使用CGLIB生成子类代理。运行时在不修改源码的情况下对目标方法进行增强。

**典型应用场景**：统一日志记录、声明式事务管理（`@Transactional`）、权限控制、性能监控、缓存处理等。

**Spring AOP vs AspectJ**：Spring AOP基于代理，仅支持方法级别拦截，是轻量级方案；AspectJ基于字节码织入，功能更强大，支持构造器、字段级别拦截。

## 99. 什么是 MySQL 的主从同步机制？它是如何实现的？

🟡 **中等** | 标签：后端, MySQL, 数据库

MySQL主从同步（复制）是一种数据冗余技术，允许将主数据库（Master）的数据变更自动同步到一个或多个从数据库（Slave）。其核心是实现**读写分离**、**数据备份**和**高可用**。

**实现流程主要分为三步：**

1.  **主库记录变更**：主库将数据变更事件（如INSERT、UPDATE）写入其**二进制日志（binlog）** 中。这是同步的源头。
2.  **从库拉取与暂存**：从库上运行的**I/O线程**连接主库，请求读取binlog。主库的**Dump线程**负责将binlog事件发送给从库。从库的I/O线程将这些事件写入本地的**中继日志（Relay Log）**。
3.  **从库重放执行**：从库的**SQL线程**读取中继日志中的事件，并逐一在本地数据库执行，从而完成数据同步。

**关键点**：默认是**异步复制**，主库提交事务后立即返回，不等待从库确认，性能高但有小概率数据丢失风险。为增强一致性，可配置**半同步复制**（主库等待至少一个从库确认接收binlog）。现代MySQL推荐使用**GTID**（全局事务标识符）来简化复制配置和故障恢复。

简单来说，主从同步就是 **“主库记录 -> 从库拉取存档 -> 从库回放”** 的流水线过程。

## 100. 什么是 Seata？

🟡 **中等** | 标签：Seata, 微服务, Spring Cloud, 后端

# Seata 概述

## 基本定义

Seata（Simple Extensible Autonomous Transaction Architecture）是阿里巴巴开源的**分布式事务解决方案**，致力于提供高性能和简单易用的分布式事务服务。它支持AT、TCC、Saga和XA等多种事务模式。

## 核心架构

Seata采用**一阶段提交+补偿**的机制，定义了三个核心组件：

| 组件 | 全称 | 职责 |
|------|------|------|
| **TC** | Transaction Coordinator | 事务协调器，独立部署，维护全局事务和分支事务状态 |
| **TM** | Transaction Manager | 事务管理器，负责开启、提交或回滚全局事务 |
| **RM** | Resource Manager | 资源管理器，负责分支事务的注册、报告和执行 |

## 工作流程

1. TM向TC申请开启全局事务，获得**全局事务ID（XID）**
2. XID在微服务调用链中传播
3. 各分支事务向TC注册并执行本地事务
4. 全部成功则TM通知TC提交；任一失败则TC通知各分支回滚

## AT模式原理

AT模式是Seata最常用的模式，其核心机制：
- **一阶段**：拦截SQL，解析并记录数据的前后镜像到undo_log表，本地事务直接提交
- **二阶段提交**：异步清理undo_log
- **二阶段回滚**：根据undo_log反向补偿数据

## 优势

- 对业务**无侵入**（AT模式）
- 与Spring Cloud、Dubbo等主流框架**无缝集成**
- 支持多种注册中心（Nacos、Eureka等）

## 101. 什么是 Spring Boot？

## 102. 什么是 Spring IOC？

🟡 **中等** | 标签：后端, Spring

Spring IOC（Inversion of Control，控制反转）是一种设计原则，其核心思想是将对象的创建、依赖管理的控制权从应用程序代码本身反转给外部容器（即Spring容器）。

具体而言，在传统开发中，对象需要自己通过`new`来创建所依赖的对象，导致代码高度耦合。而IOC模式下，对象只需声明所需的依赖，由容器在运行时自动创建并注入（Dependency Injection, DI）。这实现了组件间的解耦，使系统更灵活、易于测试和维护。

Spring通过`ApplicationContext`容器实现IOC。开发者通过XML配置或注解（如`@Component`, `@Autowired`）定义Bean及其依赖关系，容器负责初始化Bean、装配依赖并管理其完整生命周期。DI主要有构造器注入、Setter注入和字段注入等方式。

其主要优点包括：降低耦合度、提高代码可复用性、便于单元测试（可轻松替换依赖的模拟实现），以及集中管理资源（如数据源）。

因此，IOC是Spring框架的基石，它通过将控制权交给容器，彻底改变了Java应用的开发模式。

## 103. 什么是代理模式？一般用在什么场景？

🟡 **中等** | 标签：设计模式

代理模式是一种结构型设计模式，它为其他对象提供一种代理以控制对这个对象的访问。代理对象和目标对象实现相同的接口，客户端通过代理对象间接访问目标对象，在不修改目标对象的前提下，可以在代理中添加额外的功能。

**代理模式包含三个核心角色：**
- **抽象接口**：定义目标对象和代理的共同接口
- **真实对象/目标对象**：实际执行业务逻辑的对象
- **代理对象**：持有目标对象的引用，控制对其访问

**代理模式的分类：**
- **静态代理**：代理类在编译期就已确定，通过实现接口或继承来完成
- **JDK动态代理**：基于接口，通过`Proxy.newProxyInstance()`和`InvocationHandler`在运行时动态生成代理类
- **CGLIB动态代理**：基于继承，通过字节码技术生成目标类的子类，无需接口

**典型应用场景：**
1. **AOP切面编程**：如Spring AOP，实现日志记录、权限校验、事务管理
2. **远程代理**：RPC调用中，客户端调用本地代理，代理负责网络通信
3. **延迟加载**：如MyBatis的懒加载，先返回代理对象，真正使用时才查询数据库
4. **保护代理**：控制对敏感对象的访问权限
5. **缓存代理**：为开销大的操作结果提供缓存

代理模式的核心价值在于**解耦和增强**，在不修改源代码的情况下扩展对象功能，符合开闭原则。

## 104. 什么是循环依赖（常问）？

🟡 **中等** | 标签：后端, Spring

循环依赖是指两个或多个类之间相互持有对方的引用，形成类似“鸡生蛋，蛋生鸡”的闭环依赖关系。在Spring框架中，常表现为Bean A依赖Bean B，同时Bean B又依赖Bean A。

这种情况在Spring中通常可以通过**Setter注入或字段注入**自动解决，Spring通过**三级缓存机制**提前暴露一个不完整的Bean实例（早期引用），从而打破循环。但若使用**构造器注入**，则无法解决，因为对象创建时必须同时获得所有依赖，会导致启动时抛出BeanCurrentlyInCreationException异常。

循环依赖往往源于代码结构设计不合理，是**设计缺陷的信号**。最佳实践是：
1.  **尽量使用构造器注入**，让依赖关系在创建时显式明确，避免隐式循环。
2.  若遇循环依赖，应优先考虑**重构代码**，如提取公共逻辑、引入中间层或使用`@Lazy`延迟初始化。
3.  在明确了解机制的前提下，可谨慎使用Setter/字段注入解决。

总之，循环依赖是可被Spring处理的常见问题，但更应关注其背后的设计问题，力求从根本上优化代码结构。

## 105. 什么是服务熔断？

🟡 **中等** | 标签：Spring Cloud, 服务熔断, 后端, 微服务

服务熔断是微服务架构中的一种容错机制，其核心思想类似于电路中的保险丝。当某个服务因故障导致调用失败率达到预设阈值时，熔断器会自动“打开”，暂时切断对该服务的调用，避免因局部故障引发雪崩效应，从而保护整个系统的稳定性。

熔断器通常包含三种状态：**关闭**（正常调用）、**打开**（停止调用，快速失败）、**半开**（尝试部分请求以探测服务是否恢复）。当触发熔断后，经过设定的时间窗口，系统会进入半开状态，允许少量请求通过；若这些请求成功，则熔断器关闭，恢复正常调用；若仍失败，则继续维持打开状态。

在Spring Cloud中，可通过Hystrix、Resilience4j或Sentinel等组件实现服务熔断。它通过快速失败、降级处理和自动恢复，有效提升了系统的弹性和可用性，是构建高可用微服务的关键手段之一。

## 106. 什么是服务降级？

🟡 **中等** | 标签：Spring Cloud, 微服务, 后端, 服务降级

服务降级是微服务架构中的一种容错策略，指的是当系统负载过高或核心服务出现故障时，为了保障系统整体可用性，临时关闭或简化非核心服务的处理逻辑。

在Spring Cloud等微服务框架中，服务降级常通过Sentinel、Hystrix等组件实现。其典型触发场景包括：服务提供方响应超时、调用频率超出阈值、或下游服务不可用。例如，在电商系统中，当查询商品详情的API响应缓慢时，可以触发降级逻辑，返回一个包含基础信息的缓存数据或默认页面，而非让用户长时间等待。

降级的实现通常是在代码中定义一个备用的fallback方法或指定默认返回值。它不同于“熔断”：熔断是检测到故障后快速失败并切断调用链路的“开关”机制；而降级是故障发生后，系统执行的一套“备用方案”，旨在提供有损但可用的服务。

总结来说，服务降级的核心目标是通过主动舍弃非关键功能或降低服务质量，来保障系统在异常情况下的核心功能稳定与用户体验，是构建高可用分布式系统的重要手段。

## 107. 什么是物理地址，什么是逻辑地址？

🟡 **中等** | 标签：操作系统

物理地址是计算机内存硬件中存储单元的真实、唯一的地址，由内存总线直接寻址，是CPU访问内存的实际位置。

逻辑地址是程序运行时CPU生成的地址，也称为虚拟地址。它由操作系统和CPU的内存管理单元（MMU）共同管理，通常不直接对应物理硬件地址。

两者的核心区别在于：逻辑地址是程序视角下的地址空间，而物理地址是硬件视角下的实际内存位置。操作系统通过分页或分段机制，由MMU将逻辑地址动态转换为物理地址。这一过程对程序完全透明，实现了内存的抽象、保护与高效利用。在Java中，JVM进一步将逻辑地址管理起来，为对象分配堆内存，而操作系统负责最终的物理内存映射。

## 109. 什么是限流？限流算法有哪些？怎么实现的？

🟡 **中等** | 标签：后端, 系统设计, 场景题

限流是系统设计中的一种流量控制机制，其核心目的是在高并发场景下，通过限制请求速率来保护系统稳定性，防止资源耗尽导致服务不可用。

常见的限流算法主要有四种：
1.  **固定窗口计数器**：在固定时间窗口内统计请求数，超过阈值则拒绝后续请求。实现简单，但存在窗口临界点突发流量问题。
2.  **滑动窗口计数器**：将时间窗口划分为多个小格并滑动统计，流量控制更平滑，是固定窗口的优化。
3.  **漏桶算法**：请求像水一样流入桶中，以固定速率流出处理。能强制平滑流量，但无法应对突发流量。
4.  **令牌桶算法**：系统以恒定速率生成令牌放入桶中，请求需获取令牌才能被处理。在限制平均速率的同时，允许一定程度的突发流量，应用更灵活。

实现上，通常可借助开源组件快速落地。例如，在Java生态中，可以使用 **Sentinel**、**Hystrix** 等框架，或直接使用Google **Guava** 库的`RateLimiter`（基于令牌桶）。对于Web应用，在Nginx等网关层也可直接配置限流规则。核心实现思路是维护计数器或队列，并在请求入口进行判断与拦截。

## 110. 介绍一下 Reactor 线程模型？

🟡 **中等** | 标签：Netty, 后端

Reactor 线程模型是一种基于事件驱动和I/O多路复用的高性能网络编程模式，核心思想是将I/O事件的监听、分发和处理进行解耦。常见的实现模式有三种：

1. **单线程Reactor**：一个线程同时负责连接监听、I/O事件轮询和业务逻辑处理。模型简单，但一个线程处理所有任务，当某个请求处理耗时较长时，会阻塞整个系统，无法充分利用多核CPU。
2. **多线程Reactor（也称Reactor-Worker模式）**：采用一个Reactor主线程负责监听和建立连接，而将I/O读写和业务逻辑分发到由线程池管理的Worker线程中处理。这提高了并发能力，但所有连接的I/O读写仍由主线程处理，在极高并发下可能成为瓶颈。
3. **主从Reactor**：进一步分离。一个主Reactor（通常单线程）专门负责处理新连接的建立，然后将其分发给多个从Reactor（通常是线程池）。每个从Reactor负责一组连接的I/O读写和事件分发，具体的业务逻辑再交给Worker线程池处理。此模型职责划分更清晰，扩展性更强，能应对超高并发。

Netty框架的线程模型正是基于主从Reactor模式的优化实现，通过`BossGroup`和`WorkerGroup`分别对应主从Reactor，配合灵活的线程池配置，实现了高效的高并发网络通信。

## 111. 你了解的 Spring 都用到哪些设计模式？

🟡 **中等** | 标签：后端, Spring

Spring 框架的核心设计大量运用了设计模式，其中最典型的包括：

1. **工厂模式**：这是 Spring 的核心，如 `BeanFactory` 和 `ApplicationContext` 负责创建和管理 Bean 对象，实现了对象创建与使用的解耦。

2. **单例模式**：Spring Bean 默认的作用域为单例，容器保证每个 Bean 定义只创建一个实例。

3. **代理模式**：Spring AOP 的核心。通过动态代理（JDK 动态代理或 CGLIB）为目标对象创建代理，以实现声明式事务、日志等切面功能。

5. **观察者模式**：Spring 事件机制（如 `ApplicationEvent`、`ApplicationListener`）允许 Bean 之间通过事件进行松耦合的通信。

6. **适配器模式**：在 Spring MVC 的 `HandlerAdapter` 中，将不同类型的 Handler（如 Controller）适配为统一接口，便于 `DispatcherServlet` 调用。

7. **策略模式**：在资源访问（`Resource` 接口）、AOP 中实现方式的选择（JDK 或 CGLIB）等场景中灵活切换算法。

这些模式的运用使得 Spring 具备了优秀的扩展性、灵活性和可维护性。

## 112. 分布式和微服务有什么区别？

🟡 **中等** | 标签：Spring Cloud, 微服务, 后端, 分布式

**核心区别：关注点不同**

*   **分布式** 侧重于 **系统的部署和资源结构**。它指将一个应用或系统的不同模块、服务，部署到多个独立的服务器（节点）上，通过网络进行协作。其核心目标是**利用多台机器的资源提升性能、可靠性和可扩展性**，解决单机性能瓶颈问题。它是一个**技术实现方式**。

*   **微服务** 侧重于 **软件的架构和组织方式**。它是一种设计思想，强调将单一应用程序划分为一组小型、松耦合的服务。每个服务独立开发、部署和运行，围绕特定业务能力构建。其核心目标是**提高开发敏捷性、可维护性和技术栈的灵活性**。它是一种**架构风格**。

**关系与总结：**
二者并非对立，而是常结合使用。**微服务架构天然适合采用分布式部署**，即将每个微服务部署为分布式系统中的一个节点。但分布式系统不一定采用微服务架构（例如，传统SOA或单体应用的分布式部署）。

简单比喻：**分布式像“多人协作团队”，强调分工和协同完成复杂任务；微服务则像“按功能组建的多个独立小队”，每个小队自主负责一件事，它们共同组成一个“团队”。** 在现代后端开发中，常基于Spring Cloud等框架构建微服务，并将其分布式地部署。

## 113. 分布式锁一般都怎样实现？

🟡 **中等** | 标签：后端, 系统设计, 场景题

分布式锁的实现主要基于以下三种常见方案：

**1. 基于数据库实现**
最简单的方式是利用数据库的唯一索引或行级锁。例如，创建一个锁表，通过插入唯一记录来获取锁，删除记录释放锁。优点是实现简单，无需额外组件。缺点是性能差、存在单点故障风险，且锁的超时机制需要额外处理，适用于并发不高、对性能要求不严格的场景。

**2. 基于Redis实现**
这是目前最主流的方案。通常使用 `SET key value NX EX` 命令，利用其“键不存在时才设置”的原子性来实现加锁，并配合过期时间防止死锁。释放锁时需要验证value（如UUID）以确保安全。优点是性能极高。但需注意Redis集群下的主从切换可能引发锁丢失问题，可通过Redlock算法（争议较大）或Redisson客户端（封装了看门狗续期机制）来增强可靠性。

**3. 基于ZooKeeper实现**
利用ZooKeeper的临时顺序节点和Watcher机制。客户端在指定路径下创建临时顺序节点，序号最小的获得锁；其他节点监听前一个节点的状态。释放锁即删除节点，会自动触发通知。优点是强一致性，锁的释放由ZK自动管理（临时节点随会话结束而删除），可靠性高。缺点是性能相对Redis较低，且依赖ZK集群。

**总结选择**：追求高性能和简单可用选Redis；要求强一致性和高可靠性，可接受性能损失选ZooKeeper。数据库方案仅作为简易备选或理解原理。

## 114. 单例模式有哪几种实现？如何保证线程安全？

🟡 **中等** | 标签：设计模式

单例模式主要有以下四种实现方式：

1.  **饿汉式**：类加载时即创建实例，简单线程安全，但可能造成资源浪费。
2.  **懒汉式（同步方法）**：需要时创建，通过`synchronized`关键字保证线程安全，但性能较低。
3.  **双重检查锁（DCL）**：懒加载，使用`synchronized`和`volatile`关键字。`volatile`防止指令重排，`synchronized`块保证原子性，兼顾性能与安全。
4.  **静态内部类**：利用类加载机制，既懒加载又线程安全，推荐使用。
5.  **枚举**：最佳实现，枚举实例的创建由JVM保证线程安全和唯一性，还能防止反序列化和反射破坏。

**保证线程安全的核心要点**：
*   **懒加载场景**：必须进行同步控制。DCL中，`volatile`是必须的，它确保了写操作对其他线程的可见性并禁止重排序，防止获得一个未初始化的对象。
*   **加载时机控制**：饿汉式和静态内部类利用了类加载的线程安全机制（ClassLoader的加载是线程安全的），从根本上避免了多线程初始化问题。
*   **JVM保证**：枚举方式由JVM底层机制保证实例的创建是线程安全且唯一的，这是最简洁、最安全的方式。

**总结**：根据需求选择，通常推荐静态内部类或枚举实现。线程安全的本质在于控制实例的创建时机和过程的原子性、可见性、有序性。

## 115. 在什么情况下，不推荐为数据库建立索引？

🟡 **中等** | 标签：MySQL, 后端, 数据库

在以下场景中，通常不推荐为数据库建立索引：

**1. 数据表数据量很小**：当表中数据行数很少时，建立索引带来的查询性能提升微乎其微，甚至可能因为维护索引而增加开销，此时全表扫描可能更快。

**2. 数据频繁进行更新、插入或删除**：索引会降低写操作的性能。对于写密集型（OLTP）操作频繁的表，尤其是需要批量更新的场景，过多的索引会显著增加I/O和CPU开销。

**3. 列的区分度很低**：例如“性别”这类只有少数几个取值的列。为低区分度的列建立索引，效果不佳，因为索引筛选后的数据行比例仍然很大。

**4. 查询条件中不常使用该列**：索引的目的是加速查询。如果某个列几乎不出现在WHERE、JOIN或ORDER BY子句中，为其建立索引就是浪费存储和维护资源。

**5. 查询条件包含函数或表达式**：例如 `WHERE YEAR(create_time) = 2023`，对列进行函数运算会导致索引失效，此时需要创建函数索引（MySQL 8.0+支持），否则普通索引无效。

**6. 需要进行大量模糊查询**：特别是以通配符 `%` 开头的查询（如 `LIKE '%keyword'`），无法利用B+树索引。

**总结**：创建索引应遵循“按需”原则，需综合考虑表的数据量、读写比例、查询模式以及列的特征。盲目添加索引往往会适得其反。

## 116. 如何使用 MySQL 的 EXPLAIN 语句进行查询分析？

🟡 **中等** | 标签：后端, MySQL, 数据库

MySQL 的 EXPLAIN 语句是查询性能分析的核心工具，它通过模拟优化器执行SQL语句，返回其执行计划，帮助我们诊断查询性能瓶颈。

使用方法是在 SELECT 语句前添加 `EXPLAIN` 关键字。其输出结果中，以下几个字段最为关键：
1.  **type (访问类型)**：这是评估查询效率的重要指标。性能从好到差依次为：`system > const > eq_ref > ref > range > index > ALL`。应尽量避免出现 `ALL`（全表扫描）和 `index`（全索引扫描），优化至 `ref`、`range` 或更高为佳。
2.  **key (实际使用的索引)**：显示查询最终使用的索引。若为 `NULL`，则表示未命中索引，是优化的重点。
3.  **rows (预估扫描行数)**：MySQL估算找到所需记录需要读取的行数。该值越小越好，过大则需优化。
4.  **Extra (额外信息)**：包含重要提示。出现 `Using filesort` 或 `Using temporary` 通常意味着需要优化排序或分组操作。`Using index` 表示覆盖索引，效率很高。

实战中，分析 EXPLAIN 结果应聚焦于：检查 `type` 是否过于糟糕、确认 `key` 是否有效利用了索引、审视 `rows` 是否过大，以及关注 `Extra` 中的性能警示。通过分析这些字段，我们可以有针对性地调整查询、添加索引或优化表结构，从而提升数据库性能。

## 117. 如何使用 Redis 快速实现布隆过滤器？

🟡 **中等** | 标签：后端, Redis

布隆过滤器是一种空间效率高的概率型数据结构，用于判断元素是否存在。Redis通过RedisBloom模块提供原生支持。具体实现分为以下三步：

**1. 创建过滤器**  
使用`BF.RESERVE`命令初始化布隆过滤器，需指定三个关键参数：  
- 键名（如`user_filter`）  
- 误判率（如0.01，值越小精度越高但内存消耗越大）  
- 初始容量（如1000000，预估元素数量）

**2. 添加与查询元素**  
- 使用`BF.ADD`添加单个元素，或`BF.MADD`批量添加。  
- 使用`BF.EXISTS`查询单个元素，或`BF.MEXISTS`批量查询。查询结果中`1`表示可能存在（有误判概率），`0`表示绝对不存在。

**3. 注意事项**  
- **不可删除**：布隆过滤器不支持删除元素，若需删除可考虑计数布隆过滤器（Redis扩展）。  
- **空间与精度权衡**：误判率越低、容量越大，内存占用越高。  
- **分布式场景**：可为不同业务创建独立过滤器，通过集群实现横向扩展。

**示例流程**  
```bash
# 初始化
BF.RESERVE user_filter 0.01 1000000
# 添加元素
BF.ADD user_filter "user123"
# 查询存在性
BF.EXISTS user_filter "user123"  # 返回1
```
实际开发中可结合缓存或数据库使用，用布隆过滤器前置拦截无效请求，减轻后端压力。需注意首次初始化容量评估，避免频繁扩容影响性能。

## 118. 如何使用 Redis 快速实现排行榜？

🟡 **中等** | 标签：后端, Redis

**使用 Redis 实现排行榜核心方案**  
Redis 的 **Sorted Set（有序集合）** 是实现排行榜的最佳选择，因为它天然支持按分数排序和实时更新。  

**核心实现步骤：**  
1. **数据结构选择**  
   使用 `ZADD key score member` 命令，将用户ID（member）及其分数（score）存入 Sorted Set，Redis 会自动维护分数顺序。  

2. **关键操作命令**  
   - **更新分数**：`ZINCRBY key increment member`（支持原子加分，适合实时更新）。  
   - **获取排名**：`ZREVRANGE key start stop`（从高到低获取 Top N）。  
   - **查询分数及排名**：`ZSCORE key member`（查分数） + `ZREVRANK key member`（查排名，从0开始）。  

3. **相同分数处理**  
   Redis 默认按分数降序，分数相同则按 member 字典序排序。若需自定义（如按更新时间），可将时间戳融入分数（如 `score = 基础分 * 10^10 + 时间戳`）。  

4. **性能与扩展性**  
   所有操作时间复杂度为 O(log N)，支持千万级数据。若需分布式，可采用 Redis Cluster 分片存储不同排行榜。  

**注意事项**  
- 排行榜 key 建议设置过期时间或定期清理，避免内存无限增长。  
- 高频更新场景下，配合 Pipeline 批量操作提升吞吐量。  

此方案兼顾实时性、准确性与高性能，是互联网产品排行榜的通用实现方式。

## 119. 如何处理 MySQL 的主从同步延迟？

🟡 **中等** | 标签：后端, MySQL, 数据库

处理 MySQL 主从同步延迟，核心思路是定位原因并针对性优化。主要原因及解决方案如下：

**1. 优化从库与硬件配置**
*   **提升从库性能**：确保从库的硬件配置（如CPU、内存、磁盘I/O）不低于主库，避免成为瓶颈。使用SSD等高性能存储。
*   **调整从库参数**：可暂时关闭从库的某些非核心日志（如二进制日志），或调整`innodb_flush_log_at_trx_commit`等参数以提升写入性能。

**2. 减少主库压力与大事务**
*   **避免大事务**：将大的UPDATE/DELETE操作拆分成小批次执行。
*   **优化慢查询**：在主库上优化导致锁表或长时间执行的SQL。
*   **读写分离**：将耗时的统计、报表等查询转移到从库，减轻主库负载。

**3. 架构与中间件方案**
*   **并行复制**：开启从库的并行复制（如基于组提交的并行复制），能有效提升SQL线程回放效率。
*   **使用中间件**：在业务层通过中间件（如ShardingSphere）判断请求是否对数据实时性敏感。若允许延迟，则读从库；若要求强一致，则读主库。
*   **半同步复制**：可配置为半同步复制，确保至少有一个从库接收日志后再返回，但这会略微降低主库性能，适用于对数据安全要求极高的场景。

**4. 监控与排查**
使用`SHOW SLAVE STATUS`命令检查`Seconds_Behind_Master`值，并观察`Slave_SQL_Running_State`。借助`pt-heartbeat`等工具进行更精确的监控。

**总结**：应对延迟需综合监控、优化、调整架构。优先解决大事务和慢查询等根本问题，再考虑并行复制或引入中间件进行智能路由。

## 120. 如何理解 Spring Boot 中的 starter？

🟡 **中等** | 标签：Spring Boot, 后端

Spring Boot Starter 可以理解为一组**预定义的功能模块套餐**，其核心作用是简化项目的依赖管理和自动配置。

具体来说，它主要解决了两个问题：
1.  **统一依赖管理**：每个 Starter 内部聚合了实现某一功能（如Web开发、数据访问、消息队列等）所需的一组相关依赖库，并确保它们的版本相互兼容。开发者只需引入一个 Starter 依赖（例如 `spring-boot-starter-web`），而无需手动逐个添加和协调Spring MVC、Tomcat、Jackson等几十个库的版本。

使用 Starter 后，开发者通过简单的配置（如 `application.yml`）覆盖默认值，即可快速搭建功能模块，专注于业务逻辑而非基础设施。当然，开发者也可以通过 `@EnableAutoConfiguration` 或自定义 Starter 来按需定制。简而言之，Starter 是 Spring Boot “约定优于配置”理念的最佳实践，极大提升了开发效率和项目规范性。

## 121. 如何解决 Redis 中的热点 key 问题？

🟡 **中等** | 标签：后端, Redis

热点 key 是指 Redis 中被高频访问的数据 key，可能导致单节点性能瓶颈。

**解决方案主要包括：**

1.  **拆分 key**：将热点 key 拆分为多个子 key（如 key_1, key_2...），客户端根据一定规则（如随机、哈希）访问不同子 key，从而分散请求压力。这是最根本的解决方案。

2.  **本地缓存**：在应用服务器内存（如使用 Caffeine、Guava Cache）中缓存热点数据，设置较短的 TTL。请求优先读取本地缓存，未命中再访问 Redis。这能显著减少对 Redis 的访问频率。

3.  **读写分离与集群扩展**：通过 Redis 哨兵或集群模式，将读请求分散到多个从节点。配合合理的扩容，能提升整体读吞吐量。

4.  **使用 Redis 的读写分离命令**：如 `READONLY` 命令，让从节点也能处理读请求，减轻主节点压力。

5.  **设置合理的过期时间**：避免大量 key 同时过期造成缓存雪崩，可添加随机过期时间偏移量。

6.  **限流与降级**：在客户端或代理层（如使用 Sentinel）对热点 key 的访问进行限流。当检测到异常时，可执行服务降级，如返回默认值或稍后重试。

**总结**：通常结合 **“本地缓存 + key 拆分”** 作为首选方案，辅以集群读写分离和限流策略，可有效解决热点 key 问题。

## 122. 如何设计一个点赞系统？

🟡 **中等** | 标签：系统设计, 场景题

点赞系统的核心设计需兼顾**高并发写入**、**数据一致性**与**高性能读取**。以下是关键设计点：

1.  **数据模型**：通常设计两张核心表。一张记录**用户与内容的点赞关系**（如 `user_like`表，含`user_id`, `content_id`, `create_time`），用于判断“是否已赞”。另一张为**内容计数表**或缓存（如`content_count`），存储内容的实时点赞数，用于快速查询。

2.  **高并发写入处理**：针对热点内容瞬间海量点赞，采用**异步化与削峰**策略。点赞请求不直接写数据库，而是先写入Redis（记录关系和计数），再通过**消息队列（如Kafka）** 消费，批量、异步地更新数据库。这极大地保护了数据库，提升响应速度。

3.  **数据一致性保障**：在异步模式下，需保证最终一致性。Redis中的计数与数据库通过定时任务**对账**进行校准。同时，Redis点赞关系可作为缓存，减少数据库查询压力。

4.  **读取优化**：“是否点赞”状态可缓存在Redis集合中，实现O(1)查询。整体架构遵循**读写分离**，将高频读请求引流至缓存。

总结，一个健壮的点赞系统是**缓存（Redis）**、**消息队列**与**数据库**的协同，通过异步化扛住写并发，利用缓存提升读性能，并通过补偿机制确保数据最终一致。

## 123. 工厂模式和抽象工厂模式有什么区别？

🟡 **中等** | 标签：设计模式

工厂模式和抽象工厂模式都是创建型设计模式，核心区别在于**创建的对象层级和复杂度**。

**工厂模式**专注于创建**单一产品**。它通过一个工厂类，根据传入参数决定创建哪一种具体产品。其结构简单，一个工厂对应一个产品等级结构（如只生产汽车）。

**抽象工厂模式**则用于创建**一系列相关或依赖的产品族**。它提供一个抽象工厂接口，其中包含多个方法，用于创建同一产品族中的多个产品（如同时生产配套的轿车和发动机）。每个具体工厂负责生产一个产品族。

**核心区别总结**：
1.  **对象层级**：工厂模式处理单一产品等级结构；抽象工厂模式处理多个产品等级结构（产品族）。
2.  **复杂度**：抽象工厂模式更复杂，但能保证产品间的兼容性。
3.  **扩展性**：增加新产品时，工厂模式只需新增工厂类；抽象工厂模式需修改抽象工厂接口，扩展性较差，但增加新产品族则较方便。

**简单比喻**：工厂模式像一家只生产椅子的工厂；抽象工厂模式则像一家能提供整套北欧风格家具（桌子、椅子、沙发）的工厂，确保风格统一。

## 124. 数据库的脏读、不可重复读和幻读分别是什么？

🟡 **中等** | 标签：后端, MySQL, 数据库

数据库在并发事务场景下可能出现数据一致性问题，主要包括以下三种情况：

**脏读**指一个事务读取到了另一个事务尚未提交的数据。例如，事务A修改了一条记录但未提交，事务B此时读取了该修改后的值。若事务A后续回滚，则事务B读到的就是无效的“脏”数据。

**不可重复读**指同一个事务内，两次读取同一行数据，结果却不同。这通常是因为在两次读取之间，另一个事务修改了该行数据并提交。例如，事务A第一次读取余额为100元，期间事务B将其修改为50元并提交，事务A再次读取时余额变为50元，导致同一事务内读取结果不一致。

**幻读**侧重于数据条目的变化。当事务A根据条件查询时得到一批结果集，此时事务B插入或删除了符合该条件的记录并提交。当事务A再次执行相同查询时，会发现结果集中“幻影”般地多出或缺少了一些记录。

这三个问题的核心在于事务并发执行时，隔离性被破坏。通过设置合适的数据库隔离级别（如读已提交、可重复读、串行化）可以不同程度地避免它们。

## 125. 消息队列如何保证消息不丢失？

🟡 **中等** | 标签：消息队列, 后端

消息队列保证消息不丢失，需从 **生产、存储、消费** 三个环节共同保障：

1.  **生产端可靠投递**：生产者发送消息后，必须收到队列的确认（ACK）。未确认则重试或记录日志，确保消息成功投递至队列。部分队列支持事务消息或本地消息表来实现最终一致性。

2.  **队列持久化与高可用**：队列服务端收到消息后，应立即将消息**持久化到磁盘**，而不仅存于内存。同时，消息需通过**主从同步**或**多副本机制**备份到多个节点，避免单点故障导致数据丢失。

3.  **消费端可靠消费**：消费者采用**手动确认（ACK）模式**。消费者完成业务处理后，再向队列发送确认。若处理失败或网络中断，消息不会被移除，可由其他消费者重试处理，从而避免“已消费却未处理”的丢失。

4.  **配套机制**：需设计**消费者幂等性**，防止因重试导致重复消费。整体架构应具备高可用和监控告警能力，确保系统稳定。

通过上述端到端的可靠性设计，消息队列才能最大程度保证消息不丢失。

## 126. 消息队列如何保证消息的有序性（顺序性）？

🟡 **中等** | 标签：消息队列, 后端

消息队列保证消息有序性主要分为**全局有序**和**分区（局部）有序**两种实现方式：

**1. 全局有序**  
要求所有消息严格按发送顺序消费。通常通过创建**单一主题（Topic）和单一分区（Partition/Queue）** 实现，配合**单一生产者**和**单一消费者**（或消费者组内仅一个实例）。但此方式会牺牲并发性能和吞吐量，适用于低流量场景。

**2. 分区有序（最常用）**  
将业务关联的消息路由到同一分区，保证分区内有序。实现关键点：  
- **生产端**：通过**消息路由键**（如订单ID、用户ID）进行哈希取模，确保相同键的消息进入同一分区。  
- **消费端**：每个分区仅由**一个消费者单线程处理**，或使用内存队列+多线程时保证同键消息分配到同一线程。  
- **技术实现**：如Kafka通过Partition保证分区内有序，RocketMQ通过MessageQueueSharding实现类似机制。

**注意事项**：  
- 消息重试可能导致乱序，需配合**顺序重试机制**（如暂停当前队列消费）。  
- 生产端需避免异步发送导致顺序错乱。  
- 分区数量变更可能影响哈希映射，需谨慎设计。  

总结：优先采用分区有序，在业务层面通过路由键关联消息，消费端保证单分区单消费线程，兼顾性能与顺序性。

## 127. 消息队列如何处理消息堆积？

🟡 **中等** | 标签：消息队列, 后端

消息堆积是指消息队列中消息的消费速度低于生产速度，导致消息积压。处理此问题的关键在于预防、检测和应对相结合。

首先，预防是核心。通过优化消费者处理逻辑，如异步化或批量处理，提升单消费者效率；水平扩展消费者实例，增加并发能力；设置合理的队列长度阈值和监控告警，提前干预。

其次，检测机制需实时监控队列长度、消费延迟等指标，利用消息队列（如Kafka、RabbitMQ）自带工具或第三方监控系统，及时发现堆积迹象。

当堆积发生时，临时应对策略包括：动态增加消费者数量

## 128. 消息队列如何处理重复消息（保证消息的幂等性）？

🟡 **中等** | 标签：消息队列, 后端

消息队列保证幂等性，核心是确保重复消费不会产生副作用。主要方案有：

1.  **唯一消息ID + 去重表**：为每条消息分配全局唯一ID（如UUID）。消费者处理前，先查询去重表（如数据库、Redis）是否存在该ID。若不存在，执行业务逻辑并将ID存入表；若存在则跳过。这是最通用且推荐的基础方案。

2.  **数据库唯一约束/乐观锁**：若业务涉及数据更新，可在数据库表中利用唯一索引（如订单号+操作类型）防止重复插入，或通过版本号（乐观锁）进行条件更新，天然阻止重复操作。

3.  **状态机设计**：将业务实体（如订单）设计为状态机，每个状态转换是幂等的。重复消息触发同一状态转换时，因状态已变更而无法再次执行，从而保证幂等。

4.  **业务自身幂等设计**：最理想的情况是业务操作本身就是幂等的，如“设置账户余额为100元”而非“账户余额增加10元”，重复执行结果相同。

**总结**：实践中通常组合使用，如**“消息ID去重”作为第一道防线，结合“数据库唯一约束”作为最终保障**。需根据业务场景选择成本最低、可靠性最高的方案。

## 129. 消息队列设计成推消息还是拉消息？推拉模式的优缺点？

🟡 **中等** | 标签：消息队列, 后端, 场景题

消息队列的推模式和拉模式是两种核心的消息传递机制，其本质区别在于消息发送的主动权归属。推模式（Push）中，消息中间件（Broker）主动将消息发送给消费者；拉模式（Pull）中，消费者主动从Broker拉取消息。

**推模式（Push）的优缺点：**
*   **优点：** 实时性高。消息一旦到达Broker，便立即被推送给消费者，延迟极低。对于对实时性要求高的场景非常友好。
*   **缺点：** 消费者可能被大量消息淹没，难以控制消费速率，可能导致服务过载。Broker需要维护消费者状态，实现相对复杂。

**拉模式（Pull）的优缺点：**
*   **优点：** 消费者拥有完全的消费控制权，可以根据自身处理能力调整拉取频率，实现流量控制，稳定性高。实现上Broker更简单。
*   **缺点：** 实时性较低，依赖于消费者的轮询间隔，可能产生一定的消息延迟。在消息稀疏时，可能造成不必要的资源消耗。

**总结与选型：**
现代消息队列通常融合两者，如Kafka采用拉模式，但通过“长轮询”优化实时性；RocketMQ则采用推模式。选型关键取决于业务场景：对实时性、吞吐量有极致要求且消费能力稳定可选推模式；更注重稳定性、需要背压能力或流量不规律的场景则更适合拉模式。

## 131. 简单说说 Netty 的零拷贝机制？

🟡 **中等** | 标签：Netty, 后端

Netty 的零拷贝主要体现在以下三个层面：

**1. 操作系统层面的零拷贝**  
Netty 通过 NIO 的 `FileChannel.transferTo()` 方法，在底层调用操作系统的 `sendfile` 系统调用。这避免了数据在用户态和内核态之间的多次拷贝，实现文件数据直接从内核缓冲区发送到 Socket 缓冲区，减少 CPU 开销和内存拷贝。

**2. 用户态的数据整合优化**  
Netty 提供了 `CompositeByteBuf`，它可以将多个 `ByteBuf` 合并为一个逻辑视图，避免数据合并时的内存拷贝。例如，拆包场景中，只需组合已有的缓冲区，无需创建新数组复制数据。

**3. 字节操作的内存复用**  
通过 `slice()` 和 `duplicate()` 操作，Netty 可以基于已有 `ByteBuf` 创建共享内存的视图，避免拷贝。同时，Netty 使用堆外内存（DirectBuffer）配合内存池技术，减少 JVM 堆内存的分配与回收开销，进一步提升传输效率。

综上，Netty 的零拷贝核心目标是减少 CPU 拷贝次数和内存分配开销，从而显著提升网络传输性能，特别适用于高并发、大数据量的场景。

## 132. 简述 MyBatis 的插件运行原理，以及如何编写一个插件？

🟡 **中等** | 标签：后端, MyBatis

MyBatis 插件基于动态代理和责任链模式运行，允许拦截并增强四大核心对象（Executor、StatementHandler、ParameterHandler、ResultSetHandler）的方法。运行时，MyBatis 会为这些对象创建代理实例，并将配置的插件（拦截器）组成拦截链。当目标方法被调用时，会依次经过拦截链中的 `intercept` 方法，从而实现在不修改原代码的情况下进行功能扩展。

编写一个插件需遵循以下步骤：
1.  实现 `Interceptor` 接口，在 `intercept` 方法中添加自定义逻辑，并调用 `Invocation.proceed()` 以继续执行原方法。
2.  使用 `@Intercepts` 和 `@Signature` 注解，指定要拦截的目标对象及其具体方法（包括方法名和参数类型）。
3.  在 MyBatis 配置文件（如 mybatis-config.xml）中注册插件，通过 `<plugin>` 标签指定拦截器类。

插件常用于实现日志记录、分页、权限校验等通用功能。需注意，多个插件按配置逆序形成代理链，且过度使用可能影响性能及增加调试复杂度。

## 133. 线上业务服务器 CPU 飙高如何排查？

🟡 **中等** | 标签：后端, 线上问题排查

**线上服务器CPU飙高排查步骤：**

1. **快速确认与定位进程**  
   - 使用 `top` 或 `htop` 命令查看整体CPU使用率，按 `P` 键按CPU排序，确认是否为Java进程。  
   - 若为Java进程，记录其PID。

2. **分析Java线程状态**  
   - 执行 `top -Hp <PID>`，找到CPU占用最高的线程ID（TID），转换为16进制（`printf "%x\n" <TID>`）。  
   - 使用 `jstack <PID>` 导出线程堆栈，搜索对应16进制TID，定位具体线程堆栈及调用链。

3. **结合工具深入分析**  
   - 使用Arthas的 `thread -n 3` 查看最忙的线程及堆栈。  
   - 检查是否有死循环、密集计算、锁竞争（BLOCKED状态）或频繁GC（通过 `jstat -gcutil` 查看GC情况）。

4. **常见原因与优化方向**  
   - **代码问题**：死循环、正则回溯、序列化/反序列化耗时等。  
   - **资源竞争**：锁粒度过大、线程池配置不当导致线程堆积。  
   - **GC问题**：内存泄漏或堆设置不合理引发频繁Full GC，需通过GC日志分析。  
   - **外部依赖**：下游服务响应慢导致线程阻塞，需排查调用链。

5. **应急与后续**  
   - 临时重启服务或扩容缓解影响，同时定位根因优化代码、调整JVM参数或优化架构设计。

## 134. 线上发现 Redis 机器爆了，如何优化？

🟡 **中等** | 标签：后端, 场景题

当线上Redis出现资源告警（通常指内存或CPU爆满），建议按以下步骤进行优化：

**第一步：快速定位问题**
立即检查Redis监控指标，确认是内存使用率过高、连接数暴增，还是存在慢查询。使用 `redis-cli info` 或可视化工具查看实时数据，分析Key的分布情况。

**第二步：核心优化策略**
1.  **内存优化**：
    *   **优化数据结构**：避免使用大量小Key，优先使用Hash、Set等紧凑数据结构存储对象。
    *   **设置合理的过期策略**：为非永久性数据设置TTL，并采用惰性删除+定期删除策略。

## 135. 线程和进程有什么区别？

🟡 **中等** | 标签：操作系统, 计算机基础

**进程**是操作系统进行**资源分配**的基本单位，每个进程拥有独立的内存地址空间、文件描述符等系统资源，进程之间相互隔离。创建、销毁和切换进程的开销较大。

**线程**是进程内的**一个执行流**，是CPU**调度和执行**的基本单位。同一进程内的多个线程共享其资源（如内存、文件），但每个线程拥有独立的程序计数器、栈和局部变量。线程的创建、切换开销远小于进程。

核心区别在于：
1.  **资源**：进程是资源分配的单元；线程是CPU调度的单元，共享进程资源。
2.  **开销**：进程间切换开销大，需切换地址空间；线程间切换开销小。
3.  **通信**：进程间通信需通过进程间通信（IPC）机制；线程间通信可直接读写共享变量，但需注意同步。
4.  **健壮性**：一个线程崩溃可能导致整个进程终止；进程崩溃通常不会直接影响其他进程。

在Java中，一个应用程序至少有一个主线程。创建新线程可通过继承`Thread`类或实现`Runnable`/`Callable`接口来实现。

## 136. 详细描述一条 SQL 语句在 MySQL 中的执行过程。

🟡 **中等** | 标签：后端, MySQL, 数据库

一条 SQL 语句在 MySQL 中的典型执行过程如下：

**1. 连接与权限验证**
客户端首先与 MySQL 建立连接（通过 TCP/IP 或 Socket）。服务器对连接进行身份认证，并验证该用户对目标数据库及表的执行权限。

**2. 查询缓存（MySQL 8.0 已废弃）**
服务器接收到查询语句后，会先检查查询缓存。如果存在完全相同的语句及结果，则直接返回缓存结果，过程结束。否则，进入下一阶段。（注：MySQL 8.0 及之后版本已移除此功能）。

**3. 语法解析与预处理**
*   **解析器** 对 SQL 语句进行词法和语法分析，生成“解析树”。这一步主要检查语法是否正确（如关键字、括号匹配等）。
*   **预处理器** 进一步检查解析树，验证语句中涉及的表、列是否存在，解析别名，并进行权限的精确验证。

**4. 查询优化**
**优化器** 基于解析树生成一个或多个可能的执行计划。它会估算每个计划的成本（如I/O、CPU开销），并选择成本最低的计划。优化内容包括：选择合适的索引、决定表连接的顺序、重写查询等。

**5. 查询执行**
**执行器** 根据优化器选定的执行计划，调用**存储引擎** 提供的接口来获取数据。执行器会迭代处理计划中的步骤，例如，如果使用索引，可能通过引擎的“索引读”接口获取主键，再通过“主键读”获取完整行数据。

**6. 结果返回与清理**
执行器将获取到的结果数据集返回给客户端。如果启用了查询缓存（旧版本），则结果会被存入缓存。整个查询执行过程结束，资源被释放。

**核心要点总结：** 整个过程可概括为 **连接 -> 解析 -> 优化 -> 执行 -> 返回**。理解查询优化器如何基于成本选择执行计划，是理解MySQL性能优化的关键。

## 138. 说一下消息队列的模型有哪些？

🟡 **中等** | 标签：消息队列, 后端

消息队列主要有两种核心模型：

**1. 点对点模型（Queue）**
*   **核心特征**：消息发送到一个队列中，**只有一个消费者**能成功接收并处理该消息。处理完成后，消息即从队列中移除。
*   **工作方式**：支持多个消费者竞争消费，队列保证每条消息最终只被一个消费者处理。
*   **适用场景**：任务分发、负载均衡。例如，将耗时的任务（如图像处理）放入队列，由一组工作进程竞争处理，确保任务不会被重复执行。

**2. 发布-订阅模型（Topic）**
*   **核心特征**：消息发送到一个“主题”（Topic），**所有订阅了该主题的消费者**都会收到该消息的副本。
*   **工作方式**：生产者与消费者之间通过主题解耦。消费者订阅感兴趣的主题即可接收消息。
*   **适用场景**：事件广播、通知推送。例如，用户订单状态变更事件，可以广播给库存系统、积分系统、通知系统等多个服务进行处理。

**关键区别与演进**：
*   点对点模型强调**消息的竞争性消费**和**单一消费语义**。
*   发布订阅模型强调**消息的广播**和**消费的独立性**。
*   在实际的中间件（如Kafka、RocketMQ）中，发布订阅模型常结合“消费者组”概念，组内消费者对消息是竞争关系（队列模式），不同消费者组之间则是广播关系（主题模式），从而兼具两种模型的优势。

## 140. 说说 MyBatis 的缓存机制？ 

🟡 **中等** | 标签：后端, MyBatis

MyBatis 提供了一级缓存和二级缓存机制来减少数据库查询，提升性能。

**一级缓存（本地缓存）**：是SqlSession级别的缓存，默认开启且无法关闭。在同一个SqlSession中，执行相同的查询会直接返回缓存中的数据，直到该Session执行了增删改操作或手动清除缓存。其作用域局限于当前会话，不同会话间不共享。

**二级缓存（全局缓存）**：是Mapper级别的缓存，需要手动配置（在映射文件中添加`<cache/>`标签）才能开启。它作用于整个Mapper的命名空间，多个SqlSession可以共享。当一个Session提交后，数据才会进入二级缓存供其他Session使用。但它存在脏读风险，需谨慎使用于多表关联查询。

**核心要点**：一级缓存通过Session内的数据复用优化单次会话性能；二级缓存通过跨Session共享减少重复查询，但需权衡数据一致性与复杂度。缓存在更新操作或配置刷新策略时失效。

## 141. 说说 Redisson 分布式锁的原理?

🟡 **中等** | 标签：后端, Redis, Redisson

Redisson分布式锁基于Redis的原子操作实现，核心原理如下：

**1. 加锁机制**  
通过Lua脚本在Redis中执行原子操作：判断锁键是否存在，若不存在则通过`HSET`设置锁（键为线程标识，值为重入次数），并利用`PEXPIRE`设置过期时间，保证加锁与设过期的原子性。

**2. 可重入性**  
利用Redis的Hash结构记录锁的持有线程和重入次数。同一线程重复加锁时值递增，解锁时递减，直至归零才删除锁，避免死锁。

**3. Watchdog看门狗**  
后台线程定期（默认锁过期时间的1/3）检查锁是否被持有，若仍持有则自动续期，防止业务未完成锁已过期的问题。

**4. 解锁流程**  
同样通过Lua脚本原子操作：验证锁是否由当前线程持有，若重入次数为1则删除键，否则递减计数，确保只有持有者能解锁。

**5. 公平锁支持**  
通过有序集合（Sorted Set）维护等待队列，按请求顺序分配锁，避免线程饥饿。

**总结**：Redisson封装了Redis单节点/集群场景下的分布式锁实现，通过Lua脚本保证原子性，Watchdog自动续期，Hash结构支持可重入，并提供了公平锁等高级特性，简化了分布式并发控制。

## 145. 说说你知道的几种 I/O 模型

🟡 **中等** | 标签：Netty, 后端

在I/O处理中，主要存在五种模型，其核心差异在于数据等待与拷贝阶段的处理方式：

1.  **阻塞I/O（BIO）**：最传统的模型。用户线程发起系统调用后，会一直阻塞等待数据到达并从内核拷贝到用户空间，期间线程无法执行其他任务。
2.  **非阻塞I/O**：用户线程发起调用后，若数据未就绪，内核会立即返回错误，线程可轮询检查。这避免了线程被长时间挂起，但持续轮询会浪费CPU资源。
3.  **I/O多路复用**：通过`select`、`poll`或`epoll`等机制，让单个线程能同时监视多个文件描述符。当其中任一个就绪时通知应用，再进行实际读写。它用较少的线程管理大量连接，是高并发网络编程的基石，**Netty的底层核心模型即源于此**。
4.  **信号驱动I/O**：用户线程发起调用后立即返回，并通过注册信号处理函数来接收数据就绪的通知。当数据就绪时，内核发出信号，再在信号处理中拷贝数据。此模型应用较少。
5.  **异步I/O（AIO）**：真正的异步。用户线程发起调用后立即返回，由内核在数据拷贝**完成后**通知线程。整个过程无需线程主动等待或参与拷贝。

在Java语境下，BIO对应传统`Socket`编程，NIO基于I/O多路复用（对应JDK的`Channel`、`Selector`），而AIO则对应JDK7引入的`AsynchronousSocketChannel`。对于高性能网络框架，**I/O多路复用是目前最主流和高效的解决方案**。

## 146. 谈谈你了解的最常见的几种设计模式，说说他们的应用场景

🟡 **中等** | 标签：设计模式

在软件开发中，设计模式是解决特定场景下常见问题的成熟方案。以下是几种我经常使用且非常重要的设计模式：

**1. 单例模式 (Singleton Pattern)**
**核心**：确保一个类只有一个实例，并提供一个全局访问点。
**应用场景**：通常用于管理共享资源或需要统一管理的组件，例如：数据库连接池、日志记录器、Spring框架中Bean的默认作用域。实现时需注意线程安全，常见的有双重检查锁或使用枚举来实现。

**2. 工厂模式 (Factory Pattern)**
**核心**：将对象的创建过程封装起来，使客户端代码与具体类解耦。
**应用场景**：
- **简单工厂**：适用于产品种类较少且不常变动的场景，如根据参数创建不同类型的数据库连接。
- **工厂方法**：当需要扩展产品系列时，将具体产品的创建延迟到子类。例如，不同支付方式（支付宝、微信）的创建，每种支付方式对应一个工厂。

**3. 观察者模式 (Observer Pattern)**
**核心**：定义对象间一对多的依赖关系，当一个对象（主题）状态改变时，所有依赖它的对象（观察者）都会得到通知并自动更新。
**应用场景**：典型的事件驱动模型。例如，GUI按钮的点击事件监听、消息队列中的消息订阅、Spring中的事件机制（ApplicationEventPublisher）。它能有效降低耦合，实现组件间的异步通信。

除了以上三种，代理模式（用于AOP、权限控制）、策略模式（用于算法族的动态替换）等也都是面试中高频出现且实用性强的设计模式。掌握它们能显著提升代码的可维护性、可扩展性和复用性。

结合权重与随机性，性能高的服务器被选中概率更大。

**5. 最少连接算法**  
将请求发送给当前连接数最少的服务器，动态适应服务器负载变化，适用于长连接服务。

**6. 源地址哈希算法**  
根据请求IP计算哈希值，确保同一用户请求始终转发到固定服务器，可解决Session保持问题。

**7. 一致性哈希算法**  
通过哈希环实现请求分配，在服务器增减时仅影响相邻节点，适用于分布式缓存等场景。

在微服务架构中，Spring Cloud的Ribbon默认使用轮询算法，支持切换为随机、重试等策略。Nginx等中间件也集成多种算法（如加权轮询、最少连接）。实际选择需结合服务特性：无状态服务可用轮询/随机；有状态服务考虑哈希；异构集群适合加权算法；长连接场景适用最少连接算法。

## 148. Dubbo 和 Spring Cloud Gateway 有什么区别？

🔴 **困难** | 标签：后端, 分布式, Spring Cloud, Dubbo

Dubbo 与 Spring Cloud Gateway 的核心区别在于它们的定位与解决的问题不同：

1.  **本质定位**：Dubbo 是一款高性能的 **RPC（远程过程调用）框架**，核心是解决服务间（如微服务A到微服务B）的高效通信、负载均衡和容错问题。而 Spring Cloud Gateway 是一个 **API 网关**，作为系统的统一入口，主要负责外部请求的路由、过滤、限流和鉴权。

2.  **架构层次**：Dubbo 作用于**服务内部**的调用链路，专注于后端服务间的点对点通信。Gateway 作用于**系统边界**，面向客户端（如浏览器、App），是所有外部请求的第一个接入点。

3.  **功能侧重**：Dubbo 强调通信协议的灵活性（如Dubbo协议、gRPC）、高性能序列化及丰富的服务治理策略。Gateway 则侧重于请求路由、谓词/过滤器链、与Spring Cloud生态（如服务发现、配置中心）的集成，以及安全、监控等横切关注点。

4.  **技术栈**：Dubbo 是独立的框架，可与Spring等框架集成，也有自己的生态。Gateway 是 Spring Cloud 家族的一员，深度集成Spring Cloud组件。

**总结**：在微服务架构中，它们常协同工作：Gateway 作为流量总入口，将请求路由到内部服务；服务内部则通过 Dubbo 进行高性能的RPC调用。选择哪个取决于要解决的是“服务如何调用”还是“请求如何管理”的问题。

## 149. MySQL 中 如果我 select * from 一个有 1000 万行的表，内存会飙升么？

🔴 **困难** | 标签：后端, 场景题

**结论：不会导致内存飙升，但会引发严重性能问题和磁盘I/O压力。**

**核心原因：MySQL采用“流式处理”机制。**
执行 `SELECT * FROM 表` 时，MySQL服务器**不会**一次性将千万行数据全部加载到内存中。它会通过**服务器端游标**，逐行从磁盘（或内存中的缓冲池）读取数据，并通过网络**流式地**发送给客户端。客户端（如JDBC驱动）默认也会逐步获取，而非一次性全部加载到JVM堆内存。因此，主要瓶颈在于**磁盘I/O**和**网络传输**，而非服务器或客户端的内存瞬时暴涨。

**潜在风险与正确做法：**
1.  **连接阻塞与锁表**：长事务查询会持续占用数据库连接和资源，可能阻塞其他操作。
2.  **巨大的网络开销**：传输千万行数据会消耗大量带宽和时间。
3.  **客户端内存风险**：如果客户端程序（如Java应用）一次性将结果集全部存入集合（如`List`），才会导致客户端内存溢出。

**面试加分点：**
应明确指出，在生产环境中绝不应执行此类查询。正确的做法是：
- **严格分页**：使用 `LIMIT offset, size` 或基于游标的分页（如 `WHERE id > last_id`）。
- **只查必要字段**：避免 `SELECT *`，明确列出所需列以减少数据量。
- **利用索引覆盖**：确保查询条件有索引，并尽量使查询列被索引覆盖。

## 150. MySQL 中的 MVCC 是什么？

🔴 **困难** | 标签：后端, MySQL, 数据库

MVCC（多版本并发控制）是MySQL InnoDB存储引擎用于实现事务隔离和并发控制的核心机制。它通过保存数据在某个时间点的快照来实现非锁定读，从而在保证数据一致性的同时，大幅提升并发读写性能。

其实现依赖于三个核心组件：
1.  **隐藏字段**：每行数据包含两个关键字段：`DB_TRX_ID`记录最近修改该行的事务ID；`DB_ROLL_PTR`指向该行上一个版本在Undo Log中的位置。
2.  **Undo Log**：存储数据被修改前的历史版本，通过`DB_ROLL_PTR`将同一行数据的所有历史版本串联成一个**版本链**。
3.  **Read View**：事务执行快照读时，会生成一个Read View，它决定了当前事务能看到哪个版本的数据。Read View核心包含当前活跃事务ID列表。

数据可见性判断规则是：遍历版本链，根据Read View，找到第一个**对当前事务可见**的版本。这个可见性判断依赖于比较行记录的`DB_TRX_ID`与Read View中记录的活跃事务范围。

例如，在**可重复读**（Repeatable Read）隔离级别下，Read View在事务第一次读时创建，并在整个事务期间复用，从而保证了同一事务内多次读取结果一致，实现了可重复读。而在**读已提交**（Read Committed）级别下，Read View在每次读取时都会重新创建。

简而言之，MVCC通过读取历史版本避免了加锁，实现了高效的快照读，是InnoDB实现高并发事务的基石。

## 151. MySQL 是如何实现事务的？

🔴 **困难** | 标签：后端, MySQL, 数据库

MySQL的事务主要通过undo log、redo log和锁机制实现，共同保证ACID特性。

**原子性（Atomicity）** 由 **undo log** 实现。事务执行过程中，undo log会记录数据修改前的状态。若事务失败或执行回滚，数据库会利用undo log将数据恢复到事务开始前的状态，保证操作要么全部成功，要么全部撤销。

**持久性（Durability）** 由 **redo log** 实现。事务执行时，修改操作会先写入redo log并标记为已提交，再异步刷新到磁盘。即使数据页未写入磁盘时发生崩溃，重启后也能通过redo log重放已提交事务，确保数据不丢失。

**隔离性（Isolation）** 由 **锁机制** 和 **MVCC（多版本并发控制）** 实现。通过行锁、表锁等机制控制并发访问。MVCC则为每行数据创建多个版本，使读操作无需加锁，避免读写阻塞，提升并发性能。不同的隔离级别（如读已提交、可重复读）通过不同的策略组合实现。

**一致性（Consistency）** 是最终目标，依赖前三种特性共同保证。数据库通过约束检查（如主键、外键）确保数据逻辑一致，而事务管理机制则保证事务执行过程不会破坏数据一致性。

这些机制协同工作，使得MySQL能够在保证数据可靠性和正确性的前提下，高效地处理并发事务。

## 152. Redis 中跳表的实现原理是什么？

🔴 **困难** | 标签：后端, Redis

Redis跳表是一种有序数据结构，通过维护多层级链表实现快速查找。其核心思想是将链表元素抽取到更高层级，形成类似“高速公路”的索引结构，使得查找、插入和删除操作的时间复杂度可达到O(log N)。

**实现原理：**
1. **多级层状结构**：每个节点包含多个指针，分别指向不同层级的下一个节点。Redis中跳表最高层限制为32层。
2. **随机层级生成**：插入节点时，通过随机算法决定该节点的层级数，概率通常为每层1/2。例如，25%的概率达到第2层，12.5%达到第3层，依此类推。
3. **查找过程**：从最高层开始，沿当前层向右比较，若下一个节点值小于目标值则继续向右，否则向下到更底层继续查找。类似二分搜索。
4. **Redis应用场景**：主要在有序集合（Sorted Set）中使用，结合哈希表实现元素快速存取和排序操作。

**Redis优化点：**
- 节点按score排序，相同score时按字典序排列。
- 使用跳表+哈希表的双层结构，兼顾范围查询和单点查询效率。
- 通过rank值快速计算元素位置，支持范围查询操作。

跳表相比平衡树实现更简单，且范围查询效率更高，适合Redis这种对性能要求极高的场景。

## 153. 从网络角度来看，用户从输入网址到网页显示，期间发生了什么？

🔴 **困难** | 标签：网络

从输入网址到页面显示，主要经历以下网络过程：

**1. DNS域名解析**

浏览器首先检查本地缓存（浏览器缓存→系统缓存→hosts文件），若未命中则向本地DNS服务器发起递归查询，逐级向根域名服务器、顶级域名服务器、权威域名服务器查询，最终获得目标IP地址。

**2. 建立TCP连接**

通过三次握手与服务器建立TCP连接：
- 客户端发送SYN包
- 服务器回复SYN+ACK包
- 客户端发送ACK包

若为HTTPS，还需进行TLS握手，协商加密算法和密钥。

**3. 发送HTTP请求**

客户端构建HTTP请求报文，包含请求行（方法、URL、协议版本）、请求头（Host、Cookie、User-Agent等）及请求体，通过建立的TCP连接发送给服务器。

**4. 服务器处理与响应**

服务器接收请求后，经过Web服务器（如Nginx）转发至应用服务器处理，查询数据库等，最终返回HTTP响应报文（状态码、响应头、响应体）。

**5. 浏览器渲染**

浏览器解析HTML构建DOM树，解析CSS构建CSSOM树，合并生成渲染树，进行布局（Layout）计算节点位置，最后绘制（Paint）显示页面。

**6. 连接关闭**

通过四次挥手断开TCP连接（HTTP/1.1默认开启Keep-Alive可复用连接）。

此外还可能涉及CDN加速、HTTP缓存协商、重定向处理等优化机制。

## 154. 如何设计一个秒杀功能？

🔴 **困难** | 标签：后端, 系统设计, 场景题

设计秒杀功能需应对“三高”特性（高并发、高可用、高性能），核心设计思路如下：

**一、前端静态化与限流**
将商品页面静态化并缓存于CDN，降低服务器压力。前端按钮点击后置灰，防止重复提交。通过验证码或答题机制初步过滤无效请求。

**二、接入层流量控制**
使用Nginx或Spring Cloud Gateway进行限流与削峰，采用令牌桶或滑动窗口算法控制QPS。对非法请求进行IP封禁或黑名单过滤。

**三、服务层核心设计**
1. **库存预扣减**：活动开始前将库存加载至Redis，使用Lua脚本原子性地执行“判断库存 > 扣减库存”操作，避免超卖。
2. **异步下单**：Redis扣减成功后，立即通过消息队列（如RocketMQ）发送下单消息，服务端快速响应用户“排队中”，实现流量异步消化。
3. **数据库最终一致性**：消费端从队列取出消息，创建订单并扣减数据库库存，若失败可重试或进行库存回补。采用乐观锁或分布式锁保证数据一致性。

**四、数据层保障**
对热点数据使用本地缓存（如Caffeine）+分布式缓存（Redis）多级缓存。数据库层面采用分库分表、读写分离，并准备降级方案。

**五、监控与容灾**
设计独立的秒杀服务模块，与主业务解耦。实时监控Redis、数据库及队列状态，制定熔断、降级、限流等应急预案。

综上，设计核心在于**多层拦截、异步削峰、缓存抗读、队列抗写**，在保障用户体验的同时实现系统高可用。

## 155. 如果发现 Redis 内存溢出了？你会怎么做？请给出排查思路和解决方案

🔴 **困难** | 标签：后端, 场景题

当Redis出现内存溢出时，我会按以下步骤排查和解决：

**1. 快速确认与应急处理**
- 执行 `info memory` 查看 `used_memory` 和 `maxmemory`，确认是否达到上限。
- 若服务已不可用，可临时调高 `maxmemory` 或重启（注意数据持久化策略），保障业务连续性。

**2. 排查内存溢出原因**
- **数据量增长**：通过 `dbsize`、`keys *`（慎用）或扫描命令查看key数量，确认是否存在大量数据写入。
- **内存碎片**：查看 `mem_fragmentation_ratio`，若值远大于1（如>1.5），可能存在碎片。
- **大Key问题**：使用 `redis-cli --bigkeys` 扫描大String或集合，可能占用大量内存。
- **过期策略失效**：检查是否大量key未设过期时间，或 `maxmemory-policy` 配置不当（如仅为noeviction）。
- **客户端缓冲区或主从同步占用**：查看 `client-output-buffer-limit` 和复制积压缓冲区设置。

**3. 解决方案与优化**
- **数据清理**：删除冗余或临时数据，对大Key进行拆分或异步删除（如 `UNLINK`）。
- **调整配置**：
  - 设置合理的 `maxmemory` 和 `maxmemory-policy`（如 `allkeys-lru`）。
  - 优化 `timeout` 释放闲置连接，调整缓冲区大小。
- **架构优化**：
  - 启用Redis Cluster进行分片，分散内存压力。
  - 使用 `memory purge` 命令手动整理碎片（Redis 4.0+）。
- **监控预防**：设置内存使用率告警（如80%），定期分析key模式，避免缓存滥用。

## 156. 怎么分析 JVM 当前的内存占用情况？OOM 后怎么分析？

🔴 **困难** | 标签：后端, 场景题, JVM

分析JVM内存占用情况，首先可通过JDK内置工具：使用`jstat -gcutil <pid>`查看各内存区域使用率及GC活动；通过`jmap -heap <pid>`查看堆配置和使用概览；使用`jmap -dump:format=b,file=heap.hprof <pid>`生成堆转储快照，再用MAT或VisualVM分析对象占用情况。

发生OOM后，分析步骤如下：  
1. 确认OOM类型：是堆内存溢出（`java.lang.OutOfMemoryError: Java heap space`）还是元空间溢出等。  
2. 获取快照：在启动参数中添加`-XX:+HeapDumpOnOutOfMemoryError -XX:HeapDumpPath=/path`，让JVM自动生成堆转储文件。  
3. 分析堆转储：用MAT等工具查看占用内存最大的对象，检查是否有内存泄漏（如集合未清理、缓存未淘汰、静态集合累积数据）。  
4. 结合GC日志：添加`-Xloggc:gc.log -XX:+PrintGCDetails`，分析GC频率和内存回收效果。  
5. 排查代码：重点关注大对象、长生命周期对象、未关闭的资源及第三方库潜在问题。

常见原因包括内存泄漏、超大对象分配、线程栈过大或Metaspace类加载过多。解决需优化数据结构、调整堆大小、修复泄漏或增加资源限制。

## 157. 让你设计一个 RPC 框架，怎么设计？

🔴 **困难** | 标签：后端, 系统设计, 场景题

设计一个 RPC 框架，核心是实现像调用本地方法一样调用远程服务。我会从以下几个层面进行设计：

1.  **代理与桩模块**：客户端通过动态代理（如 JDK Proxy 或 CGLib）生成服务接口的代理对象，屏蔽底层网络细节。代理对象负责将方法调用信息（接口、方法、参数）封装成请求消息。
2.  **服务注册与发现**：使用独立的注册中心（如 ZooKeeper、Nacos），服务提供者启动时注册自身地址，客户端通过注册中心获取可用服务地址列表，实现服务的动态发现。
3.  **网络传输与通信**：基于高性能网络框架（如 Netty）构建通信模块，负责序列化消息（支持 Protobuf、Hessian 等）并通过 TCP 长连接进行高效传输。需处理粘包/拆包问题。
4.  **负载均衡**：客户端从注册中心获取多个服务地址后，通过负载均衡算法（如随机、轮询、加权轮询）选择一个节点发起调用。
5.  **容错与扩展性**：设计熔断、降级、重试机制保障高可用。通过 Filter 链实现可插拔的扩展点，便于统一处理日志、监控、鉴权等横切关注点。

整个流程可概括为：客户端代理发起调用 → 序列化请求 → 负载均衡选择节点 → 网络发送 → 服务端接收反序列化 → 调用本地实现 → 返回结果。设计时需重点考虑性能、容错和可扩展性。

## 158. 让你设计一个分布式 ID 发号器，怎么设计？

🔴 **困难** | 标签：后端, 系统设计, 场景题

设计分布式 ID 发号器需满足**全局唯一、趋势递增、高性能、高可用**等核心需求。常用方案如下：

**1. 方案选型**  
- **UUID**：简单但无序、较长，不适合作为数据库主键。  
- **数据库自增序列**：单点性能瓶颈，可用分段（如步长）优化，但扩展性有限。  
- **Redis 原子操作**：高性能，但需考虑持久化和集群一致性问题。  
- **雪花算法（Snowflake）**：综合优势明显，采用时间戳 + 机器 ID + 序列号结构，支持百万级 QPS，趋势递增且不依赖外部存储。

**2. 重点设计：雪花算法方案**  
- **64 位 ID 结构**：1 位符号位 + 41 位时间戳（约 69 年）+ 10 位机器 ID（支持 1024 节点）+ 12 位序列号（每毫秒 4096 个 ID）。  
- **解决时钟回拨**：通过记录上次时间戳、短暂停等待或报警机制处理。  
- **机器 ID 分配**：基于 IP、数据库或 ZooKeeper 动态分配，避免冲突。  
- **高可用设计**：服务集群化，负载均衡，并提供本地缓存应对短暂故障。

**3. 总结**  
优先选择雪花算法，需重点处理时钟回拨和机器 ID 唯一性。若业务简单，可采用数据库分段号；若追求简洁，可选用基于 Redis 的方案。

## 159. 让你设计一个短链系统，怎么设计？

🔴 **困难** | 标签：后端, 系统设计, 场景题

短链系统的核心是实现长URL到短码的映射及高效重定向。设计需考虑生成、存储、解析三大模块，以及性能、扩展性和安全性。

**1. 短码生成**：采用分布式ID生成算法（如雪花算法）生成唯一数字ID，再通过Base62编码转换为62进制字符串（如`a-z, A-Z, 0-9`），生成6-8位短码。此方法可避免哈希冲突，且单调递增便于存储优化。

**2. 存储设计**：使用关系型数据库（如MySQL）存储映射表，核心字段包括：短码（主键）、长URL、创建时间、用户ID。为提升读写性能，引入Redis缓存热点短链，采用LRU策略淘汰冷数据。

**3. 重定向服务**：接收短码请求后，优先查询Redis缓存，未命中则查询数据库并回填缓存。返回302临时重定向，便于统计点击数据；若需永久重定向且URL稳定则用301。

**4. 高可用与扩展**：
- 分布式部署短链服务，通过负载均衡分散流量。
- 数据库按短码前缀进行分库分表，支撑海量数据。
- 异步记录访问日志（如点击时间、IP、User-Agent），用于流量分析。

**5. 安全考量**：限制同一IP/用户的生成频率，防止恶意刷短链；对长URL进行黑白名单校验，避免跳转至危险网站。

**6. 扩展功能**：支持自定义短码、过期时间设置、访问统计接口等。

整体架构需保证低延迟（<10ms）、高并发（支撑万级QPS），并通过监控及时发现热点Key或异常流量。

## 160. 说一下 Kafka 中关于事务消息的实现？

🔴 **困难** | 标签：Kafka, 消息队列, 后端

Kafka 的事务消息主要用于实现跨分区、跨会话的 **“精确一次（Exactly-Once）”** 语义，保证一批消息要么全部成功提交，要么全部回滚，避免中间状态。

**核心实现机制** 依赖于一个 **事务协调器（Transaction Coordinator）**。每个生产者（Producer）都有一个唯一的事务ID，它在初始化事务时与协调器建立连接。协调器会在一个专门的内部主题 `__transaction_state` 中，持久化记录该事务的状态（开始、进行中、准备提交、已提交等）。

**事务流程** 如下：1）生产者调用 `initTransactions()` 向协调器注册事务ID。2）调用 `beginTransaction()` 后，发送的所有消息都会带上该事务ID。3）发送完成后，调用 `commitTransaction()` 或 `abortTransaction()`。4）协调器收到请求后，会先向所有涉及的分区写入“事务结束”标记，再更新事务状态为最终结果，最后通知所有消费者。

**关键前提** 是生产者必须开启 **幂等性（enable.idempotence=true）**，这能确保单分区内消息不重复。事务机制在此基础上，将保证范围扩展到了跨分区。消费者需要设置 `isolation.level=read_committed`，以确保只能读取到已提交事务的消息。事务操作会带来一定的性能开销。

## 161. 说一下 RocketMQ 中关于事务消息的实现？

🔴 **困难** | 标签：RocketMQ, 后端, 消息队列

RocketMQ 事务消息是解决分布式事务中数据最终一致性的重要机制。其核心实现可以概括为“两阶段提交”与“状态回查”相结合。

实现流程分为三个核心步骤：

1.  **发送半消息**：生产者先向 Broker 发送一条**半消息（Half Message）**，这条消息对消费者暂不可见，存储在内部的 `RMQ_SYS_TRANS_HALF_TOPIC` 队列中。发送成功后，生产者可以执行本地事务。

2.  **执行本地事务并提交**：生产者根据本地事务的执行结果，向 Broker 发送 **Commit（提交）** 或 **Rollback（回滚）** 请求。
    *   **Commit**：Broker 将半消息投递到真正的目标 Topic，消费者可见并消费。
    *   **Rollback**：Broker 将半消息标记为删除，不再投递。

3.  **事务状态回查**：如果 Broker 在一定时间内（默认60秒）未收到生产者的二次确认（Commit/Rollback），它会主动向生产者发起**回查请求**，询问本地事务的最终状态。生产者需查询本地事务结果并回复。这一步是保证最终一致性的关键，解决了生产者宕机或网络异常等问题。

通过这种“先预发、再确认、异常回查”的模式，RocketMQ 将分布式事务与本地事务绑定，保证了消息与本地事务操作的最终一致性。

## 162. 请详细描述 MySQL 的 B+ 树中查询数据的全过程

🔴 **困难** | 标签：后端, MySQL, 数据库

MySQL中查询数据时，若使用B+树索引，整个过程可分为三个关键阶段：

**1. 根节点出发**
查询从B+树的根节点开始。根节点存储了子节点的指针和键值范围，根据查询条件中的索引键值，通过二分查找确定下一个要访问的子节点。

**2. 逐层下探至叶子节点**
沿着确定的子节点指针进入下一层内部节点。内部节点同样存储键值和指针，重复二分查找过程，层层向下定位，直到到达存储实际数据指针的叶子节点。B+树的内部节点只存储索引键，不存数据，因此单个节点可容纳更多键值，使树更矮胖，减少I/O次数。

**3. 叶子节点扫描获取数据**
到达叶子节点后，所有叶子节点通过双向链表连接。在叶子节点内部，同样通过二分查找定位到具体记录。若使用聚簇索引，叶子节点直接包含完整行数据；若使用二级索引，叶子节点存储主键值，需根据主键值进行“回表”查询，再到聚簇索引中获取完整数据。

**关键点总结**：
- 查询时间复杂度稳定为O(log n)
- 磁盘I/O次数约等于树的高度（通常3-4层）
- 叶子节点双向链表优化了范围查询
- 树的分裂与平衡通过页分裂和旋转维护
