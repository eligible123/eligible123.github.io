前端对接流式输出实现AI对话效果

背景

场景

随着AI技术的普及，越来越多的客户希望在现有系统中集成AI能力（如智能问答、内容生成、数据分析等）。而流畅、实时的对话体验成为关键需求，直接影响用户对产品的满意度和使用粘性。如中冶项目创新融合了人员及项目信息智能查询等AI功能，极大提升了数据获取效率。

核心痛点：

- 响应延迟：传统API的“请求-等待-响应”模式会导致用户长时间等待，体验割裂。 - 交互不自然：非流式输出时，用户需等待AI生成完整内容后才能看到结果，失去对话的实时感。 - 性能与兼容性：不同技术方案（如SSE、WebSocket）的选型可能影响系统稳定性和开发成本。

需求目标：

- 实现流式输出：AI生成的内容以逐字或分块形式实时返回，模拟人类对话体验。 - 降低感知延迟：用户即时看到部分结果，减少等待焦虑。

在实现 AI 流式输出 时，SSE（Server-Sent Events） 和 WebSocket 是两种主流方案，但适用场景不同。以下是详细对比及选型建议

| 特性 | SSE (Server-Sent Events) | WebSocket | | | 协议 | 基于 HTTP（单向通信，服务端→客户端） | 独立协议（全双工，双向通信） | | | 数据格式 | 纯文本（text/event-stream

），需手动解析 | 支持二进制、文本，灵活性更高 | | | 连接管理 | 自动重连（内置 retry

机制） | 需手动实现断线重连 | | | 浏览器兼容性 | 除 IE 外主流浏览器均支持 | 全兼容（包括 IE 10+） | | | 适用场景 | 服务端主动推送（如AI逐字输出、实时日志） | 双向实时交互（如聊天室、游戏） | | | 复杂度 | 低（无需额外协议，直接使用 Fetch 或 EventSource） | 较高（需处理协议、心跳、状态维护） | |

综上分析，使用SSE来实现流式输出，但是当需要考虑兼容性问题的时候，可以使用WebSocket。

如果碰到控制台接口请求中，返回的数据是流式的，但是返回时间节点相同，可能是以下原因：

  • nginx需要配置 proxy_buffering:off; 关闭缓存
  • 如果前端是在本地启动的项目，由于产品使用的是webpack5的版本，默认会对响应内容进行压缩，需要在vue.config.js中配置compress: false。
