一.背景

我们先要明白：为什么要用缓存？ 如电商网站首页，每次打开都要从数据库中查询轮播图、热门商品、分类信息等数据。 如果每秒有1万个用户访问，数据库就要承受1万次查询压力。这就是典型的无缓存场景。数据库的磁盘IO速度远低于内存，当并发量上来后，系统响应变慢，数据库连接池被占满，最终导致服务不可用。

二.思路/方案

从用户请求到数据返回，数据可能经过的各级缓存路径如下图所示：

本地缓存：

在应用进程内部维护的缓存存储，数据存储在JVM堆内存中。
特点： 1）直接内存操作，无网络开销

2）无需搭建额外服务

3）每个应用实例独享自己的缓存

常用缓存： 1.Guava Cache 2.Caffeine

// Caffeine 示例（性能优于Guava Cache）

Cache caffeineCache = Caffeine.newBuilder()

    .maximumSize(10_000)

    .expireAfterWrite(10, TimeUnit.MINUTES)

    .refreshAfterWrite(1, TimeUnit.MINUTES) // 支持刷新，Guava不支持

    .recordStats()

    .build(productId -> productDao.findById(productId));

// 异步获取

public CompletableFuture getProductAsync(Long id) {

    return caffeineCache.get(id, productId -> 

        CompletableFuture.supplyAsync(() -> productDao.findById(productId)));

}

redis 缓存
- 支持多种数据类型，String / Hash / List / Set / Sorted Set 等 - 集群模式, 常见使用场景： 会话存储（分布式Session）/ 排行榜、计数器/ 分布式锁等

Memcached：
使用场景： 1. 纯缓存场景 2. 超大Value存储 3. 多线程高并发

CDN缓存：
CDN（Content Delivery Network）通过在各地部署边缘节点，将静态资源缓存到离用户最近的节点。

适用场景： 静态资源：图片、CSS、JS文件， 视频流媒体 等

还有 ：浏览器缓存， 数据库缓存 这些缓存。

三.实现过程。

在实际使用的环境中，我们需要根据情况来适当的调整我们的缓存策略。

四.总结

- 问题点

- 缓存穿透: 大量请求查询不存在的数据，绕过缓存直接击穿数据库。 - 缓存雪崩: 大量缓存同时过期，请求全部打到数据库。 - 缓存击穿: 热点Key过期瞬间，大量并发请求同时查询数据库。 - 出现以上问题的时候，我们需要调整代码的缓存刷新机制等即可尽量避免此类问题

  • 总结：

    • 缓存使用的核心原则可以总结为以下几点：

    分级缓存：合理利用多级缓存架构

    合适粒度：根据业务特点选择缓存粒度

    及时更新：设计合理的缓存更新策略

    监控告警：建立完善的缓存监控体系
