引擎二次开发指南

引擎二次开发指南

一、简介

本文档主要介绍了v1.1.0引擎二次开发中扩展包的开发相关内容，包括扩展包源码工程的目录结构，自定义扩展的开发步骤，以及扩展的调试、测试和构建打包方法。

v1.1.0扩展机制说明：扩展机制基于Java SPI规范实现，开发者通过新建扩展模块，在模块中实现云枢提供的各类SPI扩展点接口，并按照规范在META-INF/services目录下注册实现类。开发者可在扩展中使用云枢提供的OpenEngine API或Facade API进行功能开发，也能通过工具类获取自定义的Spring Service对象，最终通过打包将扩展部署到云枢服务中生效。

引入依赖时，注意以下两点：

  • 如果引用云枢相关的依赖包，需要将版本号设置为${cloudpivot.version}，以此确保在依赖包版本与cloudpivot-extensions中指定的云枢版本一致。
  • 如果引用云枢相关的、或者云枢已经依赖的第三方依赖包，则需要将<scope>设置为provided，以此确保在打包时不会将这些依赖包打入自定义扩展包中。

首先，创建MyWorkflowEventServiceImpl类，并实现WorkflowEventService接口。

然后，按照Java SPI规范注册上面的扩展实现：在resources目录下创建META-INF/services目录，并在该目录下创建一个文件，文件名必须与接口的全路径名一致，文件内容为实现类的全路径名。

各spi扩展的介绍及讲解，详见spi扩展点列表。

2.3 使用云枢提供的API

在SPI扩展点的实现、自定义Controller、Service等类中，可以直接使用云枢提供的API。云枢提供了两种API：OpenEngine API和Facade API。

首先推荐使用OpenEngine API。

2.4 使用二开自己的Spring Service对象

由于云枢引擎内部不再依赖Spring，因此在云枢SPI扩展点的实现中，无法再通过@Autowired注解来注入Spring Bean对象。

建议的方式：编写一个用于获取Spring Bean对象的工具类，然后在需要使用Spring Bean的地方，通过SpringContextUtil.getBean()方法来获取对应的Bean对象。

4. 如何构建及打包

  1. 在cloudpivot-extensions或者cloudpivot-extension-my工程下使用 mvn -U clean package 来进行构建，构建产物包含自定义扩展包jar及其引入的三方依赖包，在cloudpivot-extension工程的target目录下。
  2. 将以上构建过程中生成的自定义扩展jar包放于extensions目录，三方依赖包置于common目录中（全量覆盖模式）。
